# 腾讯云宝塔大陆部署说明

本文用于将多人端部署到腾讯云中国大陆服务器，域名示例为 `hexsss.com`，服务器通过宝塔面板运维。目标入口是：

```text
https://hexsss.com
```

队长、裁判和观众都从同一个 HTTPS 域名进入；多人端页面访问 `/api/` 和 `/health` 时由宝塔 Nginx 反向代理到本机 API 服务，不要求用户填写 `:4196` 端口。

## 上线前置

| 项目 | 要求 |
| --- | --- |
| ICP 备案 | 中国大陆服务器正式对外提供网站服务前，`hexsss.com` 需要完成 ICP 备案。备案未完成时，不建议把域名正式发给队长。 |
| 公安联网备案 | 网站正式上线后按当地要求完成公安联网备案，并把备案信息放到站点要求的位置。 |
| DNS 解析 | 在域名服务商处添加 `@` 和 `www` 的 A 记录，指向腾讯云服务器公网 IP。 |
| 安全组 | 正式上线只开放 `80` 和 `443`；`4186`、`4196` 只允许临时联调，完成反代后应关闭公网访问。 |
| HTTPS | 宝塔站点开启 SSL，建议同时开启强制 HTTPS。 |

## 宝塔环境

在宝塔面板安装：

- Nginx
- Node.js 22 或更新版本
- PM2 管理器或 Node 项目管理器

在服务器准备项目目录和数据目录：

```bash
mkdir -p /www/wwwroot
mkdir -p /www/wwwroot/hexcore-data
cd /www/wwwroot
git clone -b codex/multiplayer-realtime https://github.com/hugocat26-jpg/HEXCORE-2.0.git hexcore
cd /www/wwwroot/hexcore
npm install
```

## 进程配置

推荐用 PM2 或宝塔 Node 项目管理器启动：

```bash
npm run start:multiplayer:stack
```

正式环境变量建议：

```bash
HOST=127.0.0.1
MULTIPLAYER_APP_PORT=4186
MULTIPLAYER_API_PORT=4196
HEXCORE_SQLITE_FILE=/www/wwwroot/hexcore-data/multiplayer-store.sqlite
# PostgreSQL store 适配器接入前不要在 Node 项目里启用该项；当前仅供 schema、备份和恢复脚本使用。
# HEXCORE_POSTGRES_URL=postgres://hexcore_user:服务器内保存的密码@127.0.0.1:5432/hexcore
HEXCORE_SESSION_TTL_HOURS=48
HEXCORE_STREAM_TOKEN_TTL_SECONDS=120
```

说明：

- `HOST=127.0.0.1` 表示页面服务和 API 只监听服务器本机，由 Nginx 对外转发。
- `HEXCORE_SQLITE_FILE` 用于单机云端试运行持久化，服务重启后赛事仍可恢复。
- PostgreSQL 第一阶段已提供 schema、备份和恢复脚本；正式 store 适配器接入前，Node 项目仍使用 SQLite 作为单机部署模式。

## Nginx 反向代理

宝塔创建站点 `hexsss.com` 后，在站点配置里加入以下反向代理规则。若宝塔已生成其他 SSL 或静态配置，保留原配置，只合并 `location` 段。

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:4196/api/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
}

location = /health {
    proxy_pass http://127.0.0.1:4196/health;
    proxy_set_header Host $host;
    proxy_buffering off;
    proxy_cache off;
}

location / {
    proxy_pass http://127.0.0.1:4186/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

配置完成后重载 Nginx。

## 验证清单

1. 浏览器打开 `https://hexsss.com`，页面能进入多人房间。
2. 浏览器打开 `https://hexsss.com/health`，返回服务健康信息。
3. 裁判创建赛事后，加入页的服务地址默认显示 `https://hexsss.com`。
4. 队长使用房间码加入后，开店、购买、跳过能同步到裁判端。
5. 腾讯云安全组关闭 `4186` 和 `4196` 后，`https://hexsss.com` 仍可正常使用。
6. 宝塔重启 Node 项目后，已创建赛事能通过 SQLite 恢复。
7. 如已准备 PostgreSQL，先设置 `PGDATABASE=$HEXCORE_POSTGRES_URL`，再执行 `psql -f apps/server/postgres/schema.sql` 能创建表结构，且连接串不出现在站点页面、进程命令行或公开日志。

## 备份与回滚

每日或每场比赛结束后备份：

- `/www/wwwroot/hexcore-data/multiplayer-store.sqlite`
- PostgreSQL 试运行数据库可用 `scripts/postgres-backup.ps1` 导出自定义格式备份；连接串只从 `HEXCORE_POSTGRES_URL` 环境变量读取。
- 裁判端导出的赛事备份文件
- 当前 Git 提交号和版本号

回滚步骤：

1. 停止宝塔 Node 项目。
2. 备份当前 SQLite 文件。
3. 切换到上一个确认可用的 Git 提交或发布包。
4. 还原对应 SQLite 备份。
5. 启动 Node 项目并访问 `/health`。
6. 裁判端核对赛事、队伍和当前轮次。

## 当前限制

- 房间码明文只在创建赛事后的当前页面内存里显示；刷新后需要使用已分发的房间码加入，或重新创建赛事重新分发。
- 同步状态条和提交中状态已具备第一版，公网弱网下仍需继续观察重连体验。
- 实时订阅通道已使用短期 `streamToken` 第一版；正式赛事前仍建议继续观察代理日志、断线重连和凭据轮换策略。
- PostgreSQL schema、备份和恢复脚本已具备第一版；store 适配器接入前，单机云端部署仍优先使用 SQLite。
