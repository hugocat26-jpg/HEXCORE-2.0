# 多人端房间 UX 低保真设计

更新时间：2026-05-26

本文是 M3.2 房间 UX 设计稿，先定义信息结构、状态和安全口径。此阶段不实现页面代码；后续进入视觉稿或页面实现前，仍需按 Build Web Apps 流程产出确认版设计图并完成 Browser 验证。

## 1. 设计目标

- 普通用户只能看到“输入加入码”入口，不能从赛事 ID 反查裁判码或队长码。
- 裁判创建赛事后只在确认页看到一次完整房间码，界面必须提醒“只显示一次”。
- 裁判后续进入房间管理页时，只能看到房间码是否已发放、队伍绑定和在线状态，不再显示明文码。
- 错误提示不泄漏有效角色、有效队伍数量或邀请码存在性。
- 蓝方/本地和红方/外地语义保留给后续赛事页面；加入房间阶段只强调身份和权限。

## 2. 入口结构

```text
┌────────────────────────────────────────────────────────────┐
│ HEXCORE 2.0 多人实时系统                                  │
│                                                            │
│ 加入赛事                                                   │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ 输入加入码                                           │   │
│ │ [ ref_xxxxx / cap_xxxxx / view_xxxxx ]               │   │
│ └──────────────────────────────────────────────────────┘   │
│ [加入房间]                                                 │
│                                                            │
│ 连接状态：本地服务 / 局域网服务 / 云端服务                 │
└────────────────────────────────────────────────────────────┘
```

交互规则：

- 不提供“选择身份”按钮，身份由加入码在服务端解析。
- 加入失败统一提示“加入码无效或已失效”，不提示“这是裁判码但权限不足”等细节。
- 加入成功后根据服务端 session 跳转到裁判端、队长端或观众端。
- 前端第一版已接入该流程：无 `role` 参数且本机没有会话时显示房间加入页；加入成功后保存 sessionToken，并按服务端返回角色跳转。
- 加入成功响应必须带当前公开投影和 `stateVersion`；前端保存 sessionToken 的同时保存该版本，后续写操作以此作为 command 的 `baseVersion`。

## 3. 创建赛事后的房间码确认页

```text
┌────────────────────────────────────────────────────────────┐
│ 房间已创建：S7 班德尔保卫战                                │
│ 重要：以下房间码只显示一次，请立即分发或导出。             │
│                                                            │
│ 裁判码                                                     │
│ [ ref_a8K... ] [复制]                                      │
│                                                            │
│ 观众码                                                     │
│ [ view_m2Q... ] [复制]                                     │
│                                                            │
│ 队长码                                                     │
│ ┌──────────┬──────────────┬──────────────┐                 │
│ │ 海斗1队  │ cap_3Ka...   │ [复制]       │                 │
│ │ 海斗2队  │ cap_N1b...   │ [复制]       │                 │
│ │ ...      │ ...          │ ...          │                 │
│ └──────────┴──────────────┴──────────────┘                 │
│                                                            │
│ [我已保存，进入裁判台]                                     │
└────────────────────────────────────────────────────────────┘
```

安全规则：

- 该页只使用创建赛事响应中的一次性 `room` 数据。
- 页面刷新后不再从 `/room` 拉取明文码。
- 复制操作只写前端剪贴板，不写日志明文。
- 进入裁判台前提醒裁判保存，避免后续无法查看明文码。

## 4. 裁判房间管理摘要

```text
┌────────────────────────────────────────────────────────────┐
│ 房间管理                                                   │
│ 赛事：S7 班德尔保卫战                                      │
│                                                            │
│ 凭据状态                                                   │
│ 裁判码：已发放                                             │
│ 观众码：已发放                                             │
│ 队长入口                                                   │
│ ┌──────────┬──────────┬──────────┬──────────────┐          │
│ │ 队伍     │ 阵营     │ 码状态   │ 在线状态     │          │
│ │ 海斗1队  │ 本地     │ 已发放   │ 已连接       │          │
│ │ 海斗2队  │ 外地     │ 已发放   │ 未连接       │          │
│ └──────────┴──────────┴──────────┴──────────────┘          │
└────────────────────────────────────────────────────────────┘
```

安全规则：

- 该页调用 `GET /api/tournaments/:id/room?sessionToken=...`。
- 只有裁判、赛事管理员或超级管理员 session 可访问。
- 返回数据只包含 `codeIssued`、队伍 ID、队伍名和创建时间，不包含明文码或 hash。

## 5. 队长加入后的页面范围

队长端直接使用裁判端画面作为页面骨架，不另做一套独立抽卡 UI。导航只保留实时抽选、队伍总览、海克斯图录和我的赛程。

```text
┌────────────────────────────────────────────────────────────┐
│ 海斗1队 · 队长端                              已连接 v12   │
│ 当前阶段：第 1 轮金币商店 / 当前回合：海斗1队               │
│                                                            │
│ 实时抽选                                                   │
│ 当前商店     本队概览     可发动海克斯窗口                  │
│ [购买选手] [刷新商店] [跳过本轮]                            │
│                                                            │
│ 队伍总览 / 海克斯图录 / 我的赛程                            │
│ 均复用裁判端卡片样式，但只显示本队或只读图录信息。          │
└────────────────────────────────────────────────────────────┘
```

页面裁剪：

| 页面 | 队长端显示 | 队长端禁止 |
| --- | --- | --- |
| 实时抽选 | 与裁判端一致的当前商店、当前回合、金币、购买权、可发动海克斯提示和本队概览；自己回合保留可操作按钮；自己的海克斯选择窗口可抽取最多 5 个候选、刷新 1 张并选择 1 个。 | 撤销、修正异常、下一位、导入导出、裁判日志管理；非自己回合所有普通操作禁用；不得操作其它队伍海克斯窗口。 |
| 队伍总览 | 可查看全部队伍名称、队长、成员、阵营、费用、评分和异常提示；仅自己的队伍名称可编辑。 | 编辑其它队伍信息、补录、删除、顺位调整、设为当前、修复异常。 |
| 海克斯图录 | 显示海克斯分类、图标、触发时机、规则详情和只读详情弹窗。 | 抽取候选、刷新候选、选择、兜底分配、移除或重置海克斯。 |
| 我的赛程 | 只显示自己队伍相关场次、对手、比分、胜者和状态。 | 全局排赛、拖拽编排、录入比分、清空赛程、生成赛程和冠军管理。 |

安全规则：

- 队长端可查看全部队伍公开阵容信息，但只能编辑自己的队伍；不得查看其它队伍隐藏商店或暗信息。
- 不显示其它队伍隐藏商店、真实暗牌、房间码、裁判操作入口。
- 队长端不可撤回操作；所有写入动作仍必须经服务端校验，非自己回合只能查看当前回合队长视角。
- 队长端的开店、刷新、购买和跳过本轮必须先校验本人回合；跳过本轮可推进到下一位，但不能开放裁判端“下一位”按钮。
- 队长端可发动海克斯窗口只允许操作自己持有且未使用的海克斯；观看其它队长回合时，当前视角队长的海克斯按钮只读或隐藏。
- 进入金币商店阶段后，商店卡必须来自裁判端导入选手库和服务端权威投影；卡面展示字段沿用裁判端录入队员信息，不由前端虚构姓名、ID、费用、阵营、英雄或宣言。
- 常规商店的可见选手就是真实可购买选手；若受雪定饿的喵等显示扰乱效果影响，可见信息也只能映射到同一批导入选手，购买结算仍以服务端真实卡位为准。
- 队长端第一版写操作采用“服务端 command 先行确认 + 本地 UI 过渡渲染”：开店、刷新、购买、跳过和本队改名先提交 `/commands`，服务端通过 sessionToken、角色绑定、队伍绑定和 `stateVersion` 校验后，前端再执行当前本地动作并同步新版本号。
- 加入后的会话会订阅 `/events?view=captain|viewer`，当前先用 SSE/projection 同步公开队伍名称、改名状态、当前队伍和版本号。
- 下一阶段必须把商店、购买结果、回合推进和海克斯窗口继续改为由 SSE/projection 更新本地状态，避免多个队长端长期依赖各自 localStorage。

## 6. 观众加入后的首屏

观众端：

```text
┌────────────────────────────────────────────────────────────┐
│ HEXCORE 2.0 观众端                            只读同步中   │
│ 当前回合：海斗1队 / 第 1 轮金币商店                         │
│                                                            │
│ 当前队长视角                                               │
│ 商店公开卡     海克斯详情     队伍阵容       公开事件流     │
│ 只读查看当前回合队长可见信息，不展示隐藏卡和裁判修正细节。 │
└────────────────────────────────────────────────────────────┘
```

只读规则：

- 观众端商店展示与当前回合队长视角一致，但所有操作入口禁用或替换为只读状态。
- 观众端前端第一版使用 `role=viewer` 进入，只保留实时抽选、队伍总览和海克斯图录；队伍总览只能查看公开阵容，不显示裁判修正入口。
- 商店卡仍来自裁判端导入选手库和服务端投影，不使用虚构队员或视觉占位卡。
- 若当前回合队长受到显示扰乱，观众端只能看到该队长同样可见的导入选手映射，不额外揭示真实卡位。

## 7. 错误状态文案

| 场景 | 文案 |
| --- | --- |
| 加入码错误 | 加入码无效或已失效。 |
| 服务未连接 | 无法连接多人服务，请确认服务地址。 |
| 无权查看房间管理 | 当前身份无权查看房间管理。 |
| 会话过期 | 登录状态已失效，请重新输入加入码。 |
| 赛事不存在 | 未找到赛事，请检查服务地址或联系裁判。 |

## 8. 后续视觉要求

- 队长端优先复用裁判端深色赛事控制台风格，信息密度高但不拥挤；新增局部组件必须说明无法复用的原因。
- 队长端和观众端商店卡必须标注为“导入选手数据”，示例字段优先使用 `test-data/hexcore2_players_50.csv` 中的真实样例姓名、游戏 ID、位置、擅长英雄和宣言摘要；不得绘制虚构卡牌角色。
- 房间码确认页需要明显的“只显示一次”安全提示。
- 加入页不使用营销式 hero，不放无关说明。
- 按钮、输入框、状态点和表格行使用同一套组件。
- 所有文字在 1366px 宽度下必须不遮挡、不溢出。
- 移动端至少保证加入房间流程可用；裁判管理摘要可横向滚动或折叠。
