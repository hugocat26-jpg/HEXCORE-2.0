# 腾讯云宝塔大陆部署说明

本文用于将多人端部署到腾讯云中国大陆服务器，域名示例为 `hexsss.com`，服务器通过宝塔面板运维。目标入口是：

```text
https://hexsss.com
```

队长、裁判和观众都从同一个 HTTPS 域名进入；多人端页面访问 `/api/` 和 `/health` 时由宝塔 Nginx 反向代理到本机 API 服务，不要求用户填写 `:4196` 端口。

## 上线前置

| 项目 | 要求 |
| --- | --- |
| ICP 备案 | 中国大陆服务器正式对外提供网站服务前，`hexsss.com` 需要完成 ICP 备案。备案未完成时，不建议把域名正式发给队长。 |
| 公安联网备案 | 网站正式上线后按当地要求完成公安联网备案，并把备案信息放到站点要求的位置。 |
| DNS 解析 | 在域名服务商处添加 `@` 和 `www` 的 A 记录，指向腾讯云服务器公网 IP。 |
| 安全组 | 正式上线只开放 `80` 和 `443`；`4186`、`4196` 只允许临时联调，完成反代后应关闭公网访问。 |
| HTTPS | 宝塔站点开启 SSL，建议同时开启强制 HTTPS。 |

## 推荐部署方式

当前正式推荐为 Docker Compose + PostgreSQL 单机部署。宝塔负责站点、SSL 和 Nginx 反向代理；应用和数据库由仓库内 `docker-compose.yml` 管理。

仓库已提供脚本：

```text
deploy/baota/install-hexcore-docker.sh
deploy/baota/update-hexcore.sh
deploy/baota/backup-postgres.sh
deploy/baota/status-hexcore.sh
deploy/baota/nginx-hexcore.conf
```

快速安装：

```bash
mkdir -p /www/wwwroot
cd /www/wwwroot
git clone -b codex/multiplayer-realtime https://github.com/hugocat26-jpg/HEXCORE-2.0.git hexcore
cd /www/wwwroot/hexcore
bash deploy/baota/install-hexcore-docker.sh
```

脚本会检查 Docker、Compose、Git 和端口占用，从 `.env.example` 生成服务器 `.env`，写入随机 PostgreSQL 密码和房间码加密密钥，然后执行 `docker compose up -d --build`。脚本不会输出数据库密码或密钥，也不会开放 PostgreSQL 端口。

后续更新：

```bash
cd /www/wwwroot/hexcore
bash deploy/baota/backup-postgres.sh
bash deploy/baota/update-hexcore.sh
```

查看状态：

```bash
cd /www/wwwroot/hexcore
bash deploy/baota/status-hexcore.sh
```

## 宝塔环境

在宝塔面板安装：

- Nginx
- Docker
- Docker Compose

如果暂时不使用脚本，也可以手动准备项目目录：

```bash
mkdir -p /www/wwwroot
mkdir -p /www/wwwroot/hexcore-data
cd /www/wwwroot
git clone -b codex/multiplayer-realtime https://github.com/hugocat26-jpg/HEXCORE-2.0.git hexcore
cd /www/wwwroot/hexcore
cp .env.example .env
vi .env
docker compose up -d --build
```

`.env` 中必须设置强密码 `HEXCORE_POSTGRES_PASSWORD` 和随机 `HEXCORE_ROOM_CODE_SECRET`。该文件不得提交到 Git，也不要放到站点可访问目录之外的公开位置。

## 旧 Node/PM2 方式

旧的 Node/PM2 方式可用于临时排查，但不再作为正式默认方案。若必须临时启动：

```bash
npm install
npm run start:multiplayer:stack
```

正式环境变量建议：

```bash
HOST=127.0.0.1
MULTIPLAYER_APP_PORT=4186
MULTIPLAYER_API_PORT=4196
HEXCORE_POSTGRES_URL=postgres://hexcore_user:服务器内保存的密码@127.0.0.1:5432/hexcore
HEXCORE_SESSION_TTL_HOURS=48
HEXCORE_STREAM_TOKEN_TTL_SECONDS=120
```

说明：

- `HOST=127.0.0.1` 表示页面服务和 API 只监听服务器本机，由 Nginx 对外转发。
- `HEXCORE_POSTGRES_URL` 用于正式 PostgreSQL 持久化；不要把连接串写到宝塔站点页面、公开日志或命令截图里。

## Docker Compose 部署

服务器使用 Docker 时，可直接使用仓库内 `docker-compose.yml` 启动应用和 PostgreSQL：

```bash
cp .env.docker.example .env
vi .env
docker compose up -d --build
```

`.env` 中必须设置强密码 `HEXCORE_POSTGRES_PASSWORD` 和随机 `HEXCORE_ROOM_CODE_SECRET`，该文件不得提交到 Git，也不要放到站点可访问目录。Docker 方式启动后，宝塔 Nginx 仍按下文把站点反代到宿主机 `4186` 和 `4196`。

Docker 方式适合单机正式部署；PostgreSQL 数据保存在 `hexcore-postgres-data` volume 中，更新应用时不要执行 `docker compose down -v`。

## Nginx 反向代理

宝塔创建站点 `hexsss.com` 后，在站点配置里合并 `deploy/baota/nginx-hexcore.conf` 的反向代理规则。若宝塔已生成其他 SSL 或静态配置，保留原配置，只合并 `location` 段。

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:4196/api/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
}

location = /health {
    proxy_pass http://127.0.0.1:4196/health;
    proxy_set_header Host $host;
    proxy_buffering off;
    proxy_cache off;
}

location / {
    proxy_pass http://127.0.0.1:4186/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

配置完成后重载 Nginx。

## 验证清单

1. 浏览器打开 `https://hexsss.com`，页面能进入多人房间。
2. 浏览器打开 `https://hexsss.com/health`，返回服务健康信息。
3. 裁判创建赛事后，加入页的服务地址默认显示 `https://hexsss.com`。
4. 队长使用房间码加入后，开店、购买、跳过能同步到裁判端。
5. 腾讯云安全组关闭 `4186` 和 `4196` 后，`https://hexsss.com` 仍可正常使用。
6. 重启 `hexcore` 容器后，已创建赛事能通过 PostgreSQL 恢复。
7. `https://hexsss.com/health` 的 `runtime.storage` 显示 `postgres`，且响应中不包含连接串、数据库主机、用户名或密码。
8. PostgreSQL 模式下，`/health` 显示 `crossInstanceEventPolling` 和 `eventPollMs`。
9. 执行 `bash deploy/baota/backup-postgres.sh` 能生成 `.dump` 备份文件，且日志中不输出数据库密码。

## 备份与回滚

每日或每场比赛结束后备份：

- `/www/wwwroot/hexcore-data/multiplayer-store.sqlite`
- Docker Compose 部署时备份 PostgreSQL 数据库，不要只备份容器镜像；可使用 `scripts/postgres-backup.ps1` 或容器内 `pg_dump` 导出。
- PostgreSQL 试运行数据库可用 `scripts/postgres-backup.ps1` 导出自定义格式备份；连接串只从 `HEXCORE_POSTGRES_URL` 环境变量读取。
- 裁判端导出的赛事备份文件
- 当前 Git 提交号和版本号

回滚步骤：

1. 执行 `bash deploy/baota/backup-postgres.sh` 备份当前 PostgreSQL 数据。
2. 停止 Docker 服务：`docker compose down`，不要加 `-v`。
3. 切换到上一个确认可用的 Git 提交或发布包。
4. 如需恢复数据库，使用管理员确认过的 PostgreSQL dump 文件恢复。
5. 启动 Docker 服务并访问 `/health`。
6. 裁判端核对赛事、队伍和当前轮次。

## 当前限制

- 房间码明文只在创建赛事后的当前页面内存里显示；刷新后需要使用已分发的房间码加入，或重新创建赛事重新分发。
- 同步状态条和提交中状态已具备第一版，公网弱网下仍需继续观察重连体验。
- 实时订阅通道已使用短期 `streamToken` 第一版；正式赛事前仍建议继续观察代理日志、断线重连和凭据轮换策略。
- PostgreSQL 模式已具备轮询式跨实例 SSE 事件补偿；更高并发场景后续可升级为 WebSocket 或 PostgreSQL `LISTEN/NOTIFY`。
