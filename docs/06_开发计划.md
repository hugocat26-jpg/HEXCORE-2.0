## 1. 当前产品口径

HEXCORE 2.0 当前版本是裁判端本地选人抽卡控制台，执行模式为“裁判代执行”。

当前主流程采用阵营锁定 10 队金币商店模式：

- 10 支队伍。
- 每队 1 名队长 + 4 名队员。
- 选手分为本地、外地两个阵营。
- 本地队长只能购买或通过海克斯获得本地选手。
- 外地队长只能购买或通过海克斯获得外地选手。
- 队长选手锁定在队长池，不可被商店、补位、购买或海克斯作为队员获得。
- 队员选取以 4 轮金币商店为核心。
- 海克斯作为商店、经济、顺位、视野、补偿和特殊入队修正。
- 当前页面必须通过本地部署服务访问，不支持 `file://` 直接打开。

本地启动入口：

```powershell
npm start
```

访问地址：

```text
http://127.0.0.1:4176/
```

统一回归入口：

```powershell
npm test
```

## 2. 全局验收门禁

每完成任意功能，都必须满足以下门禁。

| 门禁 | 验收标准 |
| --- | --- |
| 回归测试 | `npm test` 通过，无新增异常用例。 |
| Diff 检查 | 本次改动文件执行 `git diff --check -- <files>` 无 trailing whitespace、冲突标记、异常 EOF。 |
| 浏览器烟测 | 内置浏览器打开 `http://127.0.0.1:4176/`，页面非空、无框架错误覆盖、console 无 error/warn。 |
| 状态保护 | 新功能不得破坏本地状态保存、导入、导出、撤销。 |
| 阵营保护 | 购买、补位、海克斯、转移、自动分配都不能造成跨阵营入队，除非产品规则显式改变。 |
| 队长保护 | 队长锁定选手不可被商店、补位、购买或海克斯作为队员获得。 |
| 容量保护 | 任意动作后队伍人数不得超过容量。 |
| 日志审计 | 影响选手、金币、顺位、卡池、海克斯状态的动作必须写事件日志。 |
| UI 可解释 | 按钮不可用时必须有明确原因，不能静默无反馈。 |
| 安全审查 | 涉及导入/导出、本地存储、静态服务、状态归一化、删除重置、依赖、命令执行、鉴权或其他信任边界时做安全审查；低风险文档、样式或钩子整理不做冗余全量审查。 |
| 推送策略 | 改动较大且验证通过后，先更新版本号，再提交并推送 GitHub；小文档改动可不立即推送。 |

## 4. 金币商店海克斯验收矩阵

### 4.0 海克斯分类口径

后续海克斯不再以青铜、白银、黄金、棱彩作为主要分类。旧 `type` 字段如仍存在，只作为历史视觉样式或兼容字段，不作为规则说明、筛选分组、验收矩阵和后续设计的主口径。

新的主分类按“玩法作用”划分，每个海克斯只能有 1 个主分类，但可以有多个辅助标签。

| 主分类 | 稳定标识 | 分类标准 | 裁判/玩家理解口径 |
| --- | --- | --- | --- |
| 商店操控类 | `shop_control` | 改变商店张数、刷新、保留、替换、费用池倾向或可见商店内容。 | 让自己更容易看到、保留或调整想要的卡。 |
| 金币运营类 | `economy` | 直接增加金币、返还金币、降低购买成本或提高高费购买能力。 | 改变资金压力和高费购买能力。 |
| 对手干扰类 | `interference` | 影响其他队长的商店、费用、金币、可见信息或决策条件。 | 让其他队长更难稳定运营。 |
| 入队替代类 | `roster_alternative` | 不通过普通商店购买完成入队、锁定入队、随机入队、自选入队或阵容重构。 | 用购买以外的方式获得或重组队员，但仍受阵营、容量、购买权约束。 |
| 顺位响应类 | `tempo_response` | 调整行动顺位、创建响应窗口、回滚购买结果或改变后续行动节奏。 | 改变谁先动、谁被延后，或在关键时点反制。 |

辅助标签用于搜索、UI 标识和测试覆盖，不替代主分类：

### 4.0.1 已完成：背水一战同阵营置换弹窗

目标：修正 `背水一战` 在金币商店模式下的规则边界，禁止跨阵营置换，并在满足发动条件时提供明显、主题化的确认弹窗，避免裁判误触后立即随机结算。

规则口径：

- 发动条件：当前队长已经拥有 4 名队员，海克斯未使用，本阵营合法候选足够；不检查购买权，已购买、已跳过或无购买权时仍可发动。
- 候选范围：全场本阵营、非禁用、非队长、非当前 4 名队员；候选可以是可选池选手，也可以是本阵营其他队伍已拥有的队员。
- 随机结算：从候选中随机抽出 4 名替代当前 4 名队员。
- 置换补偿：若抽中其他队伍已拥有的本阵营队员，该队从发动者原 4 名队员中随机获得 1 名补偿；若抽中可选池选手，则不补偿。
- 回池规则：发动者原 4 名队员中未被补偿给其他队伍的选手回到可选池。
- 顺位规则：`背水一战` 不再产生第 4 轮第一顺位效果，只作为入队替代类海克斯处理。
- 操作窗口：满员队伍默认不再进入本轮行动；但满足 `背水一战` 发动条件时必须继续保留在本轮顺位，确保裁判能看到确认弹窗并决定是否发动。背水一战本身不消耗购买权，满 4 人时购买权自然失效。

弹窗规范：

| 区块 | 标准 |
| --- | --- |
| 标题 | 显示“背水一战可发动”和当前队长名称。 |
| 数据摘要 | 展示当前队员数、本阵营候选数、可能触发置换数、可选池候选数。 |
| 当前队员 | 列出当前 4 名队员，提示这些队员会进入补偿/回池池。 |
| 候选预览 | 显示全部候选，预览框与当前队员框等高，内容超出时在框内滚动，标识来自可选池或其他队伍。 |
| 风险说明 | 明确“抽中别队队员才置换，抽中可选池不置换，未补偿原队员回可选池”。 |
| 操作 | `取消` 和 `确认发动`，确认前不改变队伍状态。 |

主题要求：

- 弹窗必须复用 `--modal-bg`、`--surface`、`--surface-strong`、`--text`、`--muted`、`--line` 等主题变量。

## 8. 多人实时系统开发计划

多人端在 GitHub 分支 `codex/multiplayer-realtime` 维护，不影响当前裁判端客户发布版。目标是把当前“裁判代执行”的本地控制台演进为“服务端权威 + 多端实时同步”的抽卡选队友系统：裁判端保留最高权限，队长端按当前回合与海克斯窗口操作，观众端只读查看当前回合队长视角。当前不建设大屏端。

### 8.1 产品目标与边界

| 方向 | 方案 | 完成口径 |
| --- | --- | --- |
| 裁判权威 | 裁判端仍是最高权限入口，可创建赛事、导入选手、修正异常、暂停流程、回滚动作和强制裁决。 | 任意队长端动作都必须经过服务端规则校验；裁判端可查看完整事件链和当前权威状态。 |
| 队长自助 | 队长登录后只看到自己的队伍、可见商店、可用海克斯、当前回合操作和必要公开信息。 | 队长不能读取其它队伍暗牌、真实身份、隐藏商店和服务端随机种子明文。 |
| 实时同步 | 裁判端、队长端和观众端共享同一抽卡房间，通过实时通道订阅不同投影。 | 任意有效动作落库后，所有在线端按权限收到增量事件；刷新页面可恢复到同一状态版本。 |
| 规则复用 | 现有 `engines/`、`core/`、`services/` 的规则边界先抽成可测试共享模块，再接入服务端。 | 现有回归用例可迁移为服务端规则测试；同一动作在本地和服务端的结果一致。 |
| 现场稳定 | 多人端支持本地局域网部署和云端部署两种模式，保留导入/导出作为应急备份。 | 网络中断后可重连；裁判可导出当前赛事快照；关键动作不因重复点击造成双重结算。 |

### 8.2 角色与权限矩阵

| 角色 | 可见范围 | 可操作范围 | 禁止事项 |
| --- | --- | --- | --- |
| `super_admin` | 全部赛事、用户和系统配置。 | 管理账号、重置赛事、配置部署参数。 | 不直接绕过规则引擎改队伍归属。 |
| `tournament_admin` | 指定赛事全部状态。 | 创建赛事、导入选手、分配裁判、发布房间。 | 不操作具体回合抽卡和比分裁决。 |
| `referee` | 当前赛事完整状态、暗信息、事件日志和异常详情。 | 执行所有裁判动作、暂停、恢复、撤销、修正、导出。 | 不通过数据库手写状态；所有修正写审计事件。 |
| `supervisor` | 当前赛事完整或接近完整状态，按配置可见暗信息。 | 只读监督、标记争议、发起复核请求。 | 不直接改变选人、商店、海克斯和比分。 |
| `captain` | 自己队伍、当前回合队长视角、本人可见商店、公开顺位、公开日志和规则提示。 | 1 次队伍改名；自己回合内抽海克斯、刷新候选、选择海克斯、刷新商店、购买、跳过；非自己回合只允许发动规则明确开放窗口的本人海克斯。 | 不操作其它队伍；非自己回合不可执行普通操作；不读取服务端隐藏字段；不越过裁判暂停。 |

### 8.3 目标架构

多人端建议从当前仓库演进为轻量 monorepo，先迁移规则核心，再拆应用壳层。

```text
apps/
  referee-web      裁判端
  captain-web      队长端
  viewer-web       观众端
  server           API、鉴权、实时网关、事件落库
packages/
  rules            当前 core 与 engines 抽出的纯规则模块
  shared           类型、常量、事件 schema、权限声明
  client-sdk       API 与实时订阅封装
```

服务端权威流程：

```text
客户端 Command
  ↓
Auth + Permission
  ↓
Idempotency + State Version 校验
  ↓
Rules Engine 计算结果
  ↓
写入 Event Store 与 Snapshot
  ↓
生成角色投影
  ↓
WebSocket 广播增量事件
```

核心原则：

- 客户端只提交意图，不提交最终结果。
- 服务端只通过规则引擎改变赛事状态。
- 所有随机由服务端生成，并记录种子摘要、随机输入、候选池摘要和结果。
- 所有多端冲突通过动作序号、幂等键和状态版本解决。
- 观众端只接收当前回合队长视角的公开投影，不接收裁判暗信息再由前端隐藏。
- 房间码、sessionToken 和角色绑定属于安全边界，不允许通过匿名接口反查或枚举；调试接口也必须按未来生产口径设计。

### 8.3.1 Codex Security + Build Web Apps 联合门禁

多人端后续按“安全边界先闭合，界面体验再扩展”的顺序推进。任何涉及登录、房间、实时通道、公开投影、队长端操作或观众端展示的阶段，都必须同时满足安全门禁和前端门禁。

| 门禁 | 执行要求 | 不通过时处理 |
| --- | --- | --- |
| 安全威胁建模 | 每个阶段开始前明确本阶段资产、攻击者输入、信任边界和不可破坏的不变量。 | 不进入代码实现，先补齐接口边界和验收项。 |
| 安全发现与验证 | 涉及 API、session、房间码、投影、导入、数据库或部署时，使用 Codex Security 做差异范围审查；发现候选问题后必须用真实接口或最小 PoC 验证。 | 可复现风险必须先修复，再继续后续 UI 或业务功能。 |
| 攻击路径复测 | 对已修复问题保留旧攻击路径回归，例如“匿名读房间码 -> 加入裁判 -> 提交裁判命令”。 | 旧攻击路径仍成功时，不得标记阶段完成。 |
| UI 复用先行 | 队长端优先直接复用当前裁判端实时抽选、队伍、海克斯和赛程页面，再按身份裁剪导航、数据范围和按钮；只有无法复用裁判端组件时，才补充局部低保真结构图。 | 未说明复用来源或新增组件原因时，不写新页面代码，不提交临时 UI。 |
| 前端浏览器验收 | UI 实现后使用内置 Browser 验证页面身份、非空、无框架错误覆盖、console 无相关 warning/error、截图证据和至少一个关键交互。 | 缺任一证据时，前端验收不得通过。 |
| 设计一致性 | 实现必须沿用裁判端既有设计系统：颜色、字体、间距、卡片半径、按钮、状态色、蓝红阵营语义、密度和响应式规则。 | 视觉漂移、文字遮挡、移动端溢出、未说明来源的新增组件都必须返工。 |

### 8.3.3 腾讯云宝塔中国大陆公网方案

当前用户已购买腾讯云中国大陆服务器，域名为 `hexsss.com`，服务器通过宝塔面板运维。公网多人端目标是让不同省份的 10 名队长直接访问 `https://hexsss.com`，在裁判主持下进入各自队长端并完成抽卡操作。该方向必须同时满足中国大陆接入合规、公网安全边界和现场操作体验。

| 方向 | 细化方案 | 验收口径 |
| --- | --- | --- |
| 大陆合规 | `hexsss.com` 在中国大陆服务器正式开放前完成 ICP 备案；网站上线后按要求完成公安联网备案；开发环境可以用公网 IP 和临时端口小范围联调，但正式分发给队长必须使用备案域名和 HTTPS。 | 部署文档明确备案、解析、安全组、宝塔站点、SSL 和公安备案步骤；正式入口不依赖裸 IP 或明文 HTTP。 |
| 同源入口 | 宝塔 Nginx 将 `https://hexsss.com/` 反代到多人端页面，将 `/api/` 和 `/health` 反代到 API 服务；前端默认 API 地址从当前页面 origin 推导，避免队长手填 `:4196`。 | 队长只需打开 `https://hexsss.com`，加入页默认服务地址正确；浏览器无跨域报错；健康检查和 API 都走同域 HTTPS。 |
| 进程守护 | 使用宝塔 Node 项目管理器、PM2 或 systemd 守护 `npm run start:multiplayer:stack`；正式启动时只监听本机 `127.0.0.1`，外网只暴露 Nginx 的 80/443。 | 服务重启后自动恢复；腾讯云安全组只需开放 80/443；`4186/4196` 不直接暴露公网。 |
| 数据持久化 | 过渡期允许 SQLite 文件持久化；正式公网赛事使用 PostgreSQL，后续可加 Redis 做在线状态和多实例广播。 | 重启后赛事、房间凭据摘要、session 摘要、事件和快照仍可恢复；备份导出可用于赛中应急。 |
| 房间分发 | 裁判创建赛事后生成“队长分发卡片”：每队一行包含访问地址、赛事 ID、队名和队长码；裁判码、观众码和队长码分开展示和下载。 | 裁判能一键复制全部、队长码、观众码，也能下载 TXT；队长拿到信息后无需理解端口和 API 地址。 |
| 旧码防误导 | 房间码明文只在创建成功后临时展示，不进入长期本地状态；刷新页面、服务重启或赛事不存在时，旧房间面板显示“此房间需要重新创建或重新加入”。 | API 中没有对应赛事时，不再展示可点击的旧裁判码；进入失败原因显示在入口页明显位置。 |
| 同步状态 | 裁判端、队长端和观众端顶部固定显示在线状态、角色、绑定队伍、当前版本、最后同步时间、session 剩余时间和重连状态。 | 队长能一眼判断“是否轮到我、是否在线、操作是否已提交”；断线后显示重连中和重新同步按钮。 |
| 公网安全 | 长期 sessionToken 不出现在 URL；SSE 先改为短期 stream token 或改用可带 `Authorization` 的流式通道，后续切换 WebSocket 鉴权通道。 | 代理日志、浏览器地址栏和下载文件不包含 sessionToken；队长端投影仍由服务端 session 绑定队伍。 |
| 操作体验 | 所有队长关键按钮提交时进入“提交中”态，重复点击被禁用；服务端拒绝时自动拉取最新投影并显示原因。 | 开店、刷新、购买、跳过和海克斯选择均有成功/失败反馈；版本冲突不会让队长停在旧画面。 |
| 裁判监控 | 裁判端增加房间状态区：当前在线队长数、离线队伍、订阅数、最新事件版本、服务健康、备案域名和备份入口。 | 裁判能在主持时发现谁没进房间、谁断线、当前服务是否可用，并可导出备份。 |

公网部署分层说明：

- P0 可联调：宝塔同源反代、默认 API origin、HTTPS 部署文档、旧码防误导、同步状态条第一版、SQLite 持久化。验收时，外省队长可通过 `https://hexsss.com` 加入房间并完成一次开店、购买和跳过，页面显示在线和最后同步时间。
- P1 可主持：房间分发卡片、短期 stream token、版本冲突自动恢复、裁判在线监控、操作提交态、备份导出提示。验收时，10 名队长远程完整演练一轮，裁判可识别离线、重连和操作拒绝原因。
- P2 正式化：PostgreSQL 存储、WebSocket 鉴权通道、凭据轮换、审计增强、部署健康面板和灾备恢复流程。验收时，可承载正式赛事，服务重启、网络波动、队长刷新页面和裁判导出备份都有明确恢复路径。

### 8.4 实时同步与并发方案

| 模块 | 方案 | 完成口径 |
| --- | --- | --- |
| 房间通道 | 每个抽卡房间建立 `tournament:{id}` 主通道，并按角色拆分 `referee`、`captain:{teamId}`、`viewer` 子通道。 | 同一事件可向不同角色发送不同投影；队长端不会收到其它队伍隐藏字段，观众端只收到当前回合队长视角公开投影。 |
| 房间码分发 | 房间码只在创建赛事响应或裁判/管理员已授权会话中展示；普通只读接口不得返回裁判码或队长码。 | 未持有有效裁判/管理员 sessionToken 时，访问房间码管理接口返回 401/403；只允许使用已知加入码调用加入接口。 |
| 动作幂等 | 每个客户端动作携带 `commandId`、`clientId`、`baseVersion`。 | 重复提交返回同一结果；版本过旧时返回可读冲突提示并触发客户端刷新。 |
| 顺序裁决 | 服务端按赛事维度串行处理会改变状态的动作；只读请求不进入写队列。 | 同一商店购买、神兵响应、轮初海克斯等窗口不会被并发双结算。 |
| 断线重连 | 客户端保存最近 `stateVersion`，重连时请求缺失事件或完整快照。 | 刷新和重连后页面恢复当前轮次、商店、海克斯窗口和赛程进度。 |
| 心跳与在线状态 | WebSocket 心跳维护在线状态，裁判端显示队长在线、离线、重连中。 | 离线队长轮到操作时，裁判端可代执行并写日志。 |
| 乐观交互 | 队长端可以显示提交中状态，但最终以服务端确认事件为准。 | 服务端拒绝动作时 UI 自动回到权威状态，并展示拒绝原因。 |
| 同步状态条 | 客户端维护 `connected`、`syncing`、`reconnecting`、`expired` 和 `offline` 状态，并展示最近成功同步时间和 `stateVersion`。 | 顶栏可见连接状态；SSE 或后续 WebSocket 断开会触发重连提示和手动重新同步入口。 |
| 公网实时鉴权 | 当前 SSE 查询参数 token 仅限本地阶段；公网阶段改为短期 stream token 或 WebSocket 握手鉴权。 | 长期 sessionToken 不出现在 URL；投影订阅失败时返回可读的过期/无权提示。 |

### 8.5 数据模型与迁移方案

多人端数据以事件流为源，快照用于快速恢复。建议核心实体如下：

| 实体 | 用途 | 关键字段 |
| --- | --- | --- |
| `tournament` | 赛事房间与规则配置。 | `id`、`name`、`mode`、`rulesVersion`、`stateVersion`、`createdBy`。 |
| `user` | 登录用户。 | `id`、`displayName`、`loginProvider`、`disabledAt`。 |
| `role_binding` | 用户在赛事中的角色绑定。 | `tournamentId`、`userId`、`role`、`teamId`。 |
| `player` | 选手库。 | `playerId`、`name`、`camp`、`score`、`poolCost`、`lockedAsCaptain`。 |
| `team` | 队伍与阵容。 | `teamId`、`name`、`camp`、`captainId`、`coins`、`rosterSlots`。 |
| `hexcore_assignment` | 海克斯归属和使用状态。 | `teamId`、`hexcoreId`、`usageState`、`roundLocks`。 |
| `draft_session` | 抽选流程状态。 | `round`、`turnIndex`、`currentTeamId`、`shopState`、`hexWindows`。 |
| `shop_offer` | 当前商店卡槽。 | `slotId`、`realPlayerId`、`displayPlayerId`、`cost`、`purchased`；`realPlayerId` 和 `displayPlayerId` 均必须指向本场导入选手库，常规商店二者一致，雪定饿的喵等扰乱效果只改变可见映射，不创造新选手。 |
| `match` | 赛程和比分。 | `matchId`、`round`、`sideA`、`sideB`、`scoreA`、`scoreB`、`winnerId`。 |
| `event` | 权威动作日志。 | `eventSeq`、`type`、`actorId`、`payload`、`createdAt`、`stateVersion`。 |
| `snapshot` | 状态快照。 | `tournamentId`、`stateVersion`、`stateJson`、`checksum`。 |
| `audit_log` | 管理和安全审计。 | `actorId`、`action`、`target`、`ipHash`、`userAgentHash`。 |
| `room_credential` | 房间加入凭据。 | `tournamentId`、`role`、`teamId`、`codeHash`、`createdAt`、`disabledAt`、`lastUsedAt`。 |
| `session` | 临时登录会话。 | `sessionTokenHash`、`tournamentId`、`userId`、`role`、`teamId`、`issuedAt`、`expiresAt`、`revokedAt`。 |

旧版本导入策略：

- 保留当前 JSON 导入为“裁判端快照导入”入口。
- 导入时先执行 normalize 和 integrity 检查，生成迁移预览，不直接发布到房间。
- 迁移完成后写入 `StateImported` 事件，并为原始文件生成 checksum。
- 旧版本中缺少的多人字段使用默认角色绑定和系统生成 ID 补齐。
- 导入后的首个快照必须能再次导出，并可被当前裁判端阅读为应急备份。

### 8.7 关键业务流程

| 流程 | 多人端设计 | 验收口径 |
| --- | --- | --- |
| 创建赛事 | 管理员创建赛事，选择规则版本，导入选手和队伍，裁判确认发布房间。 | 发布前可预览异常；发布后生成房间码和角色入口。 |
| 队长抽海克斯 | 服务端为指定队长生成 5 候选，队长可刷新其中 1 张，最终选择 1 张。 | 刷新和选择动作有幂等保护；队长只能操作自己的候选会话。 |
| 轮初海克斯 | 服务端在开店前统一检查启元、大炮等轮初窗口，裁判端和相关队长端同步提示。 | 轮初窗口未关闭前不能生成商店；加速之门窗口不被免费开店跳过。 |
| 金币商店 | 服务端根据当前顺位、概率、卡池和海克斯效果生成商店；队长端只提交购买意图。 | 购买后服务端完成扣费、入队、购买权、海浪/血雾等组合结算。 |
| 响应窗口 | 神兵天降、海浪等响应效果由服务端生成窗口，按顺位和触发规则裁决。 | 多名队长同时响应同一购买时，只能有一个权威结果。 |
| 异常修正 | 裁判可暂停赛事，执行修正动作，服务端写入审计事件并广播新快照。 | 修正前后都有差异摘要；队长端收到可读说明。 |
| 赛程比分 | 当前多人端抽卡阶段不迁移赛程比分；赛程相关能力保留在裁判端维护版。 | 多人端权限和抽卡主流程稳定后再单独评估是否需要迁移。 |

### 8.8 技术选型建议

| 领域 | 建议 | 原因 |
| --- | --- | --- |
| 服务端语言 | Node.js + TypeScript。 | 复用当前 JavaScript 规则资产，降低迁移成本，并用类型约束多人端事件。 |
| API 框架 | Fastify 或 Hono。 | 路由清晰、插件生态成熟，适合权限中间件和 schema 校验。 |
| 实时协议 | WebSocket，优先封装房间、重连和事件确认；是否使用 Socket.IO 由部署复杂度决定。 | 需要低延迟广播、断线恢复和多角色投影。 |
| 数据库 | PostgreSQL。 | 适合赛事、事件、快照、用户和审计数据；便于赛后复盘。 |
| 缓存与队列 | 首版可用进程内队列；多人并发增强后引入 Redis。 | 先保证单赛事串行动作正确，再扩展多实例部署。 |
| 前端框架 | 多端拆分后可引入 Vite + React；规则核心先迁移，UI 框架后迁移。 | 当前原生前端已稳定，先拆规则可降低一次性重构风险。 |
| Schema 校验 | 使用 Zod 或同类 schema 工具描述 Command、Event、Snapshot。 | 服务端、客户端和测试共享同一份协议定义。 |
| 测试 | 保留 `npm test`，新增规则包测试、API 合约测试、WebSocket 多客户端模拟和权限测试。 | 多人端风险主要来自并发、权限、重连和投影泄漏。 |

### 8.9 多人端验收门禁

多人端每个阶段除继续通过当前 `npm test` 与 `git diff --check` 外，还需要补充以下门禁：

| 门禁 | 验收口径 |
| --- | --- |
| 规则一致性 | 现有金币商店、海克斯、赛程和导入导出回归迁移到共享规则测试，关键路径输出一致。 |
| 权限隔离 | 每个角色都有正向和拒绝用例；队长端不能越权读写其它队伍或暗信息。 |
| 实时恢复 | 多客户端模拟覆盖连接、断线、重连、重复命令、过期版本和服务端拒绝。 |
| 投影检查 | 公开 payload 不包含真实隐藏身份、内部随机输入、审计敏感字段和管理入口。 |
| 商店来源 | 队长端和观众端商店卡只能渲染服务端投影中的导入选手字段；前端不得编造选手姓名、ID、阵营、费用或英雄信息。 |
| 随机审计 | 抽卡、海克斯、补位和随机置换均记录可复盘摘要；赛后可从事件流重放。 |
| 数据迁移 | 当前导出的本地 JSON 可导入多人端并生成快照；迁移过程输出异常报告。 |
| 浏览器验证 | 裁判端、队长端、观众端至少各完成一次非空页面、console、关键交互验证。 |
| 视觉继承 | 队长端、观众端和多人裁判面板必须复用当前裁判端深色控制台风格、状态色、卡片密度和按钮语义；设计稿和实现都要说明继承关系。 |
| 安全审查 | 引入鉴权、WebSocket、数据库、导入迁移、部署脚本或依赖变更时，使用 Codex Security 做差异范围审查。 |

### 8.10 多人端规则冲突裁决储备

当前阶段仅记录规则分析，不进入本轮实现闭环。若后续将海克斯分配规则调整为“每名队长最多 1 个海克斯、同名海克斯可重复、共 10 名队长”，需要重点复核跨队长执行冲突；同一队长内部因为只持有 1 个海克斯，基本不会出现自有海克斯互相抢窗口的问题。

| 冲突点 | 影响范围 | 储备裁决方向 |
| --- | --- | --- |
| 多名队长影响同一商店 | 阵营封锁、抬价干扰、骤雨血雾清风、雪定饿的喵 | 数值型效果可叠加；显示扰乱类同一商店建议只保留一次有效扰乱，并在面板解释覆盖来源。 |
| 多名队长锁定同一选手 | 和我困在一起 | 可以同时锁定；结算时按触发顺序实时校验，目标已离开卡池则后续锁定失效并写日志。 |
| 多名队长响应同一次购买 | 神兵天降 | 按当前顺位开放响应，第一名成功夺取后关闭本次响应窗口，其它持有者不可再抢同一购买。 |
| 多名队长拥有轮初顺位技 | 神秘贤者·启元、大炮已充能 | 先结算启元，再按当前顺位逐个询问大炮；大炮仍不得越过启元保护。 |
| 多名队长拥有海浪 | 海浪，我没吃饭 | 每轮只随机 1 名持有者触发海浪状态；未触发者本轮不获得海浪免疫和夺取判定。 |
| 多个直接入队效果竞争卡池 | 稳健补强、盲盒、质变、分解、背水一战、和我困在一起 | 按执行时点实时校验卡池、阵营、队长保护、容量和重复归属；先成功者占用目标，后续目标不足则按各自规则失败或降档。 |

总体裁决储备口径：同名可重复只改变分配池，不改变单次执行窗口；所有跨队长冲突以触发窗口、当前顺位和实时目标校验为准。该项仅作为后续规则改造索引，当前版本不调整执行逻辑。

### 8.11 多人端开发约束

- `main` 保持裁判端维护和客户发布；多人端工作集中在 `codex/multiplayer-realtime`。
- 本机多人端 Git worktree 路径为 `E:\only_why\HEXCORE2.0\multiplayer`，跟踪 `codex/multiplayer-realtime`；同级 `hex-core2.0` 继续作为裁判端仓库，不得覆盖。
- 每次多人端开发完成后，直接在 `E:\only_why\HEXCORE2.0\multiplayer` 运行门禁、提交并推送 `codex/multiplayer-realtime`，不再使用普通目录同步流程。
- 新增服务端后，客户端不得直接信任本地状态、URL 参数或前端隐藏字段。
- 多人端先做规则包和服务端权威，再做队长端美化；不要先复制一套不可复用 UI。
- 多人端 UI 实现前必须先产出设计图或低保真结构图，经确认后再写页面代码；未确认的临时 UI 不纳入提交。
- 用户已确认队长端优先直接复用裁判端页面；该方向下不再等待独立视觉稿，改为在现有裁判页面上做权限和数据裁剪。
- 队长端、观众端和多人裁判面板必须以当前裁判端为视觉基线；新增 UI 要先映射到现有色板、面板、按钮、选手卡、海克斯卡、状态胶囊和日志组件，再决定是否新增组件。
- 商店卡和入队结果必须引用本场由裁判端导入的选手库；多人端 UI、测试夹具和视觉稿不得把队员当作虚构卡牌生成。
- 所有 Command、Event、Snapshot 需要版本号，破坏性变更必须提供迁移函数。
- localStorage 在多人端只保存登录态、偏好和最近房间，不作为赛事权威数据源。
- 任何导入、鉴权、实时通道、数据库和部署脚本改动，都要先过本地门禁，再按风险使用 Codex Security。
- M3.1 完成前不得继续开发队长端、观众端 UI；避免把已知权限绕过固化进前端流程。
