# 开发计划

## 1. 当前产品口径

HEXCORE 2.0 当前版本是裁判端本地选人抽卡控制台，执行模式为“裁判代执行”。

当前主流程采用阵营锁定 10 队金币商店模式：

- 10 支队伍。
- 每队 1 名队长 + 4 名队员。
- 选手分为本地、外地两个阵营。
- 本地队长只能购买或通过海克斯获得本地选手。
- 外地队长只能购买或通过海克斯获得外地选手。
- 队长选手锁定在队长池，不可被商店、补位、购买或海克斯作为队员获得。
- 队员选取以 4 轮金币商店为核心。
- 海克斯作为商店、经济、顺位、视野、补偿和特殊入队修正。
- 当前页面必须通过本地部署服务访问，不支持 `file://` 直接打开。

本地启动入口：

```powershell
npm start
```

访问地址：

```text
http://127.0.0.1:4176/
```

统一回归入口：

```powershell
npm test
```

## 2. 全局验收门禁

每完成任意功能，都必须满足以下门禁。

| 门禁 | 验收标准 |
| --- | --- |
| 回归测试 | `npm test` 通过，无新增失败用例。 |
| Diff 检查 | 本次改动文件执行 `git diff --check -- <files>` 无 trailing whitespace、冲突标记、异常 EOF。 |
| 浏览器烟测 | 内置浏览器打开 `http://127.0.0.1:4176/`，页面非空、无框架错误覆盖、console 无 error/warn。 |
| 状态保护 | 新功能不得破坏本地状态保存、导入、导出、撤销。 |
| 阵营保护 | 购买、补位、海克斯、转移、自动分配都不能造成跨阵营入队，除非产品规则显式改变。 |
| 队长保护 | 队长锁定选手不可被商店、补位、购买或海克斯作为队员获得。 |
| 容量保护 | 任意动作后队伍人数不得超过容量。 |
| 日志审计 | 影响选手、金币、顺位、卡池、海克斯状态的动作必须写事件日志。 |
| UI 可解释 | 按钮不可用时必须有明确原因，不能静默失败。 |
| 安全审查 | 改动较大时做差异范围安全审查；计划完成时做全量安全审查。 |
| 推送策略 | 改动较大且验证通过后推送 Gitee；小文档改动可不立即推送。 |

### 2.1 技能执行要求

后续执行本开发计划时，必须按任务类型加载对应技能，并把技能产出的验证结果写入执行记录：

- 涉及前端、交互、浏览器验证、UI 状态、布局和视觉改造的计划项，必须先加载 Build Web Apps 相关技能。
- 针对已渲染页面的修复、调试和交互验证，优先使用 `build-web-apps:frontend-testing-debugging`；如 Browser 插件可用，必须按该技能要求使用内置 Browser 完成页面身份、非空页面、错误覆盖层、console、截图和交互证明。
- 涉及新界面、重设计、视觉风格或完整前端体验搭建的计划项，使用 Build Web Apps 的前端构建/设计流程，并在桌面和必要移动视口完成渲染验证。
- 涉及代码改动的计划项，在本地回归和差异检查通过后，必须使用 Codex Security 做代码安全审查；较大改动做差异范围审查，计划全部完成或准备推送阶段做全量审查。
- Codex Security 审查发现可成立问题时，必须先修复并重新运行验证；不得在存在未处理安全发现时标记计划完成或推送 Gitee。
- 每轮技能执行结果必须至少记录：加载的技能、验证命令、浏览器证据或失败原因、安全审查结论、剩余风险。

### 3.5.1 已完成：海克斯五抽一流程

目标：替换当前“每次 3 候选、选满 3 个”的旧流程，改为“每名队长 5 候选、可刷新其中 1 张、最多选 1 张”。

| 功能 | 验收标准 |
| --- | --- |
| 候选数量 | 点击“抽取海克斯”后，当前队长最多生成 5 张候选海克斯；全局剩余可用候选不足 5 张时，只展示可抽取数量。 |
| 抽取失败 | 全局剩余可用未占用海克斯为 0 时，本次抽取失败并提示“全局剩余可用海克斯不足 1个”。 |
| 候选去重 | 候选中不能重复；不能出现规则已禁用海克斯；不能出现该队长已持有海克斯；不能出现其他队长已选择海克斯。 |
| 单张刷新 | 每名队长本次抽取期间只能刷新 1 张候选；刷新按钮位于候选卡片内；刷新后该候选位替换为新的合法海克斯。 |
| 刷新约束 | 刷新出的海克斯不能与当前候选重复，不能与该队长已持有海克斯重复，不能与其他队长已选择海克斯重复，不能是已禁用海克斯。 |
| 刷新次数显示 | UI 显示“可刷新 1/1”或“已刷新 1/1”；使用后刷新按钮禁用。 |
| 选择上限 | 队长最多选择 1 张海克斯；点击确认后该海克斯加入队长持有列表，本次 5 张候选会话关闭。 |
| 全局唯一 | 海克斯被任一队长确认选择或裁判兜底分配后，立即进入全局占用池，不再进入后续其他队长候选池；选择和兜底分配均会二次校验全局占用。 |

| 手动下一位 | 当前队长选择完成后，不自动跳下一位；裁判必须点击“下一位”切换。 |
| 已选状态 | 已完成五抽一的队长在抽取顺序中显示“已选 1/1”；未完成显示“待选”。 |
| 重抽保护 | 已完成五抽一的队长再次点击抽取时应提示“该队长已完成海克斯选择”，不能覆盖已选结果。 |
| 取消会话 | 裁判可取消当前 5 张候选；取消后不写入持有海克斯，事件日志记录取消。 |

| 导入兼容 | 旧状态中已有 3 个海克斯的队长保持已持有记录；新流程按至少 1 个海克斯视为已完成，后续未完成队长按 1/1 执行。 |

## 4. 金币商店海克斯验收矩阵

### 4.0 海克斯分类口径

后续海克斯不再以青铜、白银、黄金、棱彩作为主要分类。旧 `type` 字段如仍存在，只作为历史视觉样式或兼容字段，不作为规则说明、筛选分组、验收矩阵和后续设计的主口径。

新的主分类按“玩法作用”划分，每个海克斯只能有 1 个主分类，但可以有多个辅助标签。

| 主分类 | 稳定标识 | 分类标准 | 裁判/玩家理解口径 |
| --- | --- | --- | --- |
| 商店操控类 | `shop_control` | 改变商店张数、刷新、保留、替换、费用池倾向或可见商店内容。 | 让自己更容易看到、保留或调整想要的卡。 |
| 金币运营类 | `economy` | 直接增加金币、返还金币、降低购买成本或提高高费购买能力。 | 改变资金压力和高费购买能力。 |
| 对手干扰类 | `interference` | 影响其他队长的商店、费用、金币、可见信息或决策条件。 | 让其他队长更难稳定运营。 |
| 入队替代类 | `roster_alternative` | 不通过普通商店购买完成入队、锁定入队、随机入队、自选入队或阵容重构。 | 用购买以外的方式获得或重组队员，但仍受阵营、容量、购买权约束。 |
| 顺位响应类 | `tempo_response` | 调整行动顺位、创建响应窗口、回滚购买结果或改变后续行动节奏。 | 改变谁先动、谁被延后，或在关键时点反制。 |

辅助标签用于搜索、UI 标识和测试覆盖，不替代主分类：

| 标签维度 | 可选标签示例 | 用途 |
| --- | --- | --- |
| 触发时机 | 获得时、开店前、商店打开后、购买前、购买后、轮次开始、下一轮结算、响应窗口 | 决定按钮出现时机和执行队列排序。 |
| 目标对象 | 自己、商店卡、选手、其他队长、连续队长、全局队伍 | 决定目标选择控件和合法性校验。 |
| 资源影响 | 金币、刷新、商店数量、商店质量、购买权、顺位、信息可见性、队员入队 | 决定日志、冲突解释和风险提示。 |
| 风险属性 | 稳定收益、随机收益、延迟收益、对手干扰、反制窗口、阵容重构 | 决定回归测试优先级和裁判提示强度。 |
| 规则敏感 | 消耗购买权、直接入队、可回滚、可延迟、每局一次、每轮一次、被动自动 | 决定是否需要二次确认和状态回滚测试。 |

| 用户系统 | 支持裁判、裁判监督、队长角色；权限边界明确。 |
| 队长登录抽海克斯 | 队长只能操作自己的五抽一会话；裁判可监督和回滚。 |
| 队长登录选卡 | 队长只能在自己回合选择；裁判可暂停、恢复、强制裁决。 |
| WebSocket 同步 | 多端状态实时同步；断线重连后可恢复当前状态。 |
| 服务端权威抽卡 | 抽卡随机由服务端执行；客户端不能篡改结果。 |
| 可验证随机 | 每次随机有种子或审计记录，可赛后复盘。 |
