## 阶段 1：项目骨架

目标：建立新架构目录，不迁移旧业务逻辑。

任务：

- 引入 TypeScript 和 Vite。
- 建立 `domain` 类型。
- 建立 `store` 和事件日志结构。
- 建立 `engines` 空实现和测试骨架。
- 保留旧 `src` 原型文件作为参考或迁移来源。

验收：

- 项目可启动。
- TypeScript 类型检查通过。
- 无金币相关类型。
- `Player.cost` 不出现在新代码中。

## 阶段 2：核心引擎

目标：先实现纯业务规则，不做复杂 UI。

任务：

- Draft Engine 状态机。
- Turn Order Engine 顺位计算。
- Pool Engine 卡池映射。
- Probability Engine 抽卡。
- Assignment Engine 入队和自动分配。
- Visibility Engine 可见性视图。

验收：

- 单元测试覆盖基础轮抽。
- 单元测试覆盖舞会女王、恶魔契约、启元、致盲。
- 可以通过事件 replay 得到相同状态。

## 阶段 3：海克斯自动化

目标：实现主要海克斯效果。

优先级：

1. 低风险规则：质变、巨人杀手、舞会女王、恶魔契约、稳扎稳打。
2. 顺位规则：启元、开饭啦、顺位互换、背水一战。
3. 暗信息规则：致盲吹箭、雪定饿的喵。
4. 复杂规则：锁定契约、地狱三头犬、盲盒、摄影艺术家。
5. 信息增强：知识来源于分解。

验收：

- 每个海克斯至少有一条自动化测试。
- 冲突类海克斯有顺位解释。
- 队伍满员后自动跳过。

## 阶段 4：裁判工作台 UI

目标：让裁判可以完成完整赛事流程。

任务：

- 队长设置。
- 海克斯分配。
- 轮抽控制台。
- 当前回合操作面板。
- 事件日志面板。
- 数据备份和恢复。

验收：

- 裁判可以完整完成 12 队 4 轮抽卡。
- 页面刷新后可恢复状态。
- 能导出完整 JSON。
