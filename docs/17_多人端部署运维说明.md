# 多人端部署运维说明

本文记录 `codex/multiplayer-realtime` 分支当前多人端的本地和局域网运行方式。当前版本默认使用内存存储，适合现场规则联调；需要跨重启保留赛事时可启用本地 JSON 文件、SQLite 数据库存储或 PostgreSQL 正式持久化。腾讯云中国大陆 + 宝塔 + `hexsss.com` 的正式公网部署见 [docs/18_腾讯云宝塔大陆部署说明.md](18_腾讯云宝塔大陆部署说明.md)。

## 启动方式

| 场景 | 命令 | 默认地址 |
| --- | --- | --- |
| 仅启动多人端页面 | `npm run start:multiplayer` | `http://127.0.0.1:4186/` |
| 仅启动多人端 API | `npm run start:server` | `http://127.0.0.1:4196/health` |
| 同时启动页面和 API | `npm run start:multiplayer:stack` | 页面 `4186`，API `4196` |

局域网试运行时，在服务端电脑设置：

```powershell
$env:HOST="0.0.0.0"
npm run start:multiplayer:stack
```

队长和观众用服务端电脑的局域网 IP 访问页面，例如 `http://192.168.1.23:4186/`；加入页里的 API 地址填写 `http://192.168.1.23:4196`。

公网部署时不要让队长直接访问 `4186` 或 `4196`。宝塔 Nginx 应把 `https://hexsss.com/` 反代到页面服务，把 `/api/` 和 `/health` 反代到 API 服务；此时加入页会默认使用 `https://hexsss.com` 作为服务地址。

## 环境变量

| 变量 | 默认值 | 用途 |
| --- | --- | --- |
| `HOST` | `127.0.0.1` | 监听地址；局域网访问使用 `0.0.0.0`。 |
| `MULTIPLAYER_APP_PORT` | `4186` | 多人端静态页面端口。 |
| `MULTIPLAYER_API_PORT` | `4196` | 多人端 API 端口。 |
| `MULTIPLAYER_APP_ROOT` | `apps/multiplayer` | 多人端静态文件根目录。 |
| `HEXCORE_DATA_FILE` | 空 | 可选。填写 JSON 文件路径后，API 会把赛事、房间凭据摘要和 session 持久化到该文件。 |
| `HEXCORE_SQLITE_FILE` | 空 | 可选。填写 SQLite 文件路径后，API 会把赛事、房间凭据摘要和 session 持久化到本地数据库；优先级高于 `HEXCORE_DATA_FILE`。 |
| `HEXCORE_POSTGRES_URL` | 空 | PostgreSQL 正式持久化连接串。设置后 API 使用 PostgreSQL store；不要写入命令参数、日志、前端状态或导出文件。 |
| `HEXCORE_SESSION_TTL_HOURS` | `24` | session 有效期小时数；过期后需要重新输入加入码进入房间。 |
| `HEXCORE_STREAM_TOKEN_TTL_SECONDS` | `120` | 队长 SSE 实时订阅短期凭据有效期秒数；长期 sessionToken 不进入事件流 URL。 |

## 本地持久化

需要服务重启后继续使用同一赛事时，启动 API 前设置：

```powershell
$env:HEXCORE_DATA_FILE="E:\hexcore-data\multiplayer-store.json"
npm run start:multiplayer:stack
```

启用后，`/health` 的 `runtime.storage` 会显示 `memory+file`。持久化文件包含服务端权威状态、私有选手池、事件、房间凭据摘要和 session 摘要；不保存创建赛事时一次性展示的房间码明文。该文件应只由裁判或服务器管理员保管，不应发给队长或观众。

## SQLite 数据库存储

需要更接近正式数据库的本地运行方式时，启动 API 前设置：

```powershell
$env:HEXCORE_SQLITE_FILE="E:\hexcore-data\multiplayer-store.sqlite"
npm run start:multiplayer:stack
```

启用后，`/health` 的 `runtime.storage` 会显示 `memory+sqlite`，但不会公开本机数据库文件路径。SQLite 存储不引入外部依赖，使用 Node.js 自带 `node:sqlite`，表内只保存 `tournaments`、`roomAccess` 和 `sessions` 三类 JSON 快照；不保存创建赛事时一次性展示的房间码明文。该模式用于本机或单机局域网试运行，不适合作为多实例云端共享数据库。

## PostgreSQL 正式持久化准备

已提供 PostgreSQL store、schema 和备份恢复脚本：

```text
apps/server/postgres-store.js
apps/server/postgres/schema.sql
scripts/postgres-backup.ps1
scripts/postgres-restore.ps1
```

初始化数据库时在服务器环境变量中设置连接串，然后用 `psql` 应用 schema：

```powershell
$env:HEXCORE_POSTGRES_URL="postgres://hexcore_user:只在服务器保存的密码@127.0.0.1:5432/hexcore"
$env:PGDATABASE=$env:HEXCORE_POSTGRES_URL
psql -f apps/server/postgres/schema.sql
```

备份：

```powershell
$env:HEXCORE_POSTGRES_URL="postgres://hexcore_user:只在服务器保存的密码@127.0.0.1:5432/hexcore"
powershell -ExecutionPolicy Bypass -File scripts/postgres-backup.ps1 -OutputPath E:\hexcore-data\backup.dump
```

恢复：

```powershell
$env:HEXCORE_POSTGRES_URL="postgres://hexcore_user:只在服务器保存的密码@127.0.0.1:5432/hexcore"
powershell -ExecutionPolicy Bypass -File scripts/postgres-restore.ps1 -BackupPath E:\hexcore-data\backup.dump -Clean
```

安全要求：

- `HEXCORE_POSTGRES_URL` 只能存在于服务器环境变量或 PM2 受控配置中，不写入 Git、前端页面、本地状态、URL 查询参数或公开日志。
- PostgreSQL 表只保存房间凭据摘要和 session 摘要，不保存创建赛事后一次性展示的房间码明文或 sessionToken 明文。
- 备份文件包含服务端权威状态、凭据摘要、审计和检查点，按裁判/管理员私密备份处理。
- HTTP store 适配器使用 Node `pg` 连接池访问 PostgreSQL；请求链路不能用阻塞式外部 `psql` 命令临时代替。

启动：

```powershell
$env:HEXCORE_POSTGRES_URL="postgres://hexcore_user:只在服务器保存的密码@127.0.0.1:5432/hexcore"
npm run start:multiplayer:stack
```

启用后，`/health` 的 `runtime.storage` 会显示 `postgres`，但不会公开连接串、数据库主机、用户名或密码。

## Docker 本机演示

客户电脑已安装 Docker Desktop 时，可用 Compose 同时启动页面、API 和 PostgreSQL：

```powershell
Copy-Item .env.docker.example .env
notepad .env
docker compose up -d --build
```

`.env` 至少要设置 `HEXCORE_POSTGRES_PASSWORD`。该文件已被 `.gitignore` 排除，不要提交正式密码。

启动后访问：

```text
页面：http://127.0.0.1:4186/
健康检查：http://127.0.0.1:4196/health
```

Compose 结构：

| 服务 | 说明 |
| --- | --- |
| `hexcore` | Node 应用容器，同时启动多人端页面服务和 API 服务。 |
| `postgres` | PostgreSQL 16 容器，数据写入 `hexcore-postgres-data` 命名 volume。 |

常用命令：

```powershell
docker compose ps
docker compose logs -f hexcore
docker compose down
docker compose down -v
```

`docker compose down -v` 会删除 PostgreSQL 数据卷，只能在确认不要保留赛事数据时使用。

## 健康检查

访问：

```text
GET /health
```

返回内容包含服务名、规则版本、启动时间、运行秒数、存储状态、赛事数量、房间数量、session 有效期秒数和 SSE 订阅数量。该接口不返回房间码、sessionToken、队伍暗信息或完整赛事状态。

## session 过期

队长、裁判和观众通过加入码进入房间后，服务端会签发本机会话并返回 `expiresAt`。默认有效期为 24 小时；过期后原页面不能继续读取队长投影、提交操作或管理房间，需要重新输入加入码。现场长时间赛程可在启动前设置 `HEXCORE_SESSION_TTL_HOURS`，例如：

```powershell
$env:HEXCORE_SESSION_TTL_HOURS="48"
npm run start:multiplayer:stack
```

## 备份导出

裁判或管理员可导出权威赛事备份：

```text
GET /api/tournaments/:id/export
Authorization: Bearer 裁判或管理员会话
```

返回 `backupVersion`、`exportedAt`、`checksum` 和完整权威赛事状态。该备份包含私有选手池、事件、审计和检查点，只能交给裁判或管理员保管，不应发给队长或观众。

匿名访问或把 sessionToken 放在 URL 查询参数中会返回 `401`，观众访问会返回 `403`。导出内容不包含房间码明文、房间码摘要或 sessionToken。

## 当前限制

- 默认存储为进程内内存，服务重启后赛事会丢失；重要试运行建议使用 Docker Compose/PostgreSQL，或设置 `HEXCORE_POSTGRES_URL`、`HEXCORE_SQLITE_FILE`、`HEXCORE_DATA_FILE`，并定期导出备份。
- 房间码明文只在创建赛事后的当前页面内存里显示，不会随完整状态长期保存；刷新后需要使用已分发的房间码加入，或重新创建赛事重新分发。
- 当前 SSE 为第一版实时通道，队长订阅使用短期 `streamToken`，支持刷新恢复快照，但还未做跨实例事件补偿。
- 云端多实例部署前仍需补跨实例事件补偿；HTTPS 反向代理第一版见 `docs/18_腾讯云宝塔大陆部署说明.md`。
