# HEXCORE 2.0 项目总览

更新时间：2026-05-30

当前状态：裁判端维护版保持稳定；多人端在 `codex/multiplayer-realtime` 分支持续开发，现行计划以 [../planning/当前待开发计划.md](../planning/当前待开发计划.md) 为准，固定门禁见 [../planning/开发门禁.md](../planning/开发门禁.md)。现行规则以 [阵营锁定10队金币商店模式实施规范.md](阵营锁定10队金币商店模式实施规范.md) 和最新用户指南为准。

## 1. 项目定位

HEXCORE 2.0 是面向线下或线上主持场景的选人抽卡控制系统。当前稳定形态是裁判端本地控制台；多人端正在演进为“裁判端最高权限、队长端按回合操作、观众端只读观看当前回合队长视角”的服务端权威抽卡系统。

术语边界：

- 本系统处理“队长通过金币商店和海克斯完成选人”的流程，不处理实际对局内操作。
- 当前系统内的“比赛”“赛程”只指选人结束后的阵营对抗框架和比分记录。
- 当前不建设大屏端；多人端只包含裁判端、队长端和观众端。

## 2. 当前阶段目标

当前裁判端目标是保持可本地运行、可审计、可回归验证：

- 固定 10 队，每队 5 人，队长占用 1 个名额。
- 支持本地人、外地人双阵营，队长和队员必须同阵营。
- 支持 1-5 费金币商店，四轮抽选，每次商店 5 张卡。
- 支持初始金币、轮收入、刷新费用、折扣、返还和海克斯经济效果。
- 支持海克斯五抽一、PNG 图标、详情弹窗、执行队列和响应窗口。
- 支持赛程生成、首轮拖拽、确认轮空、比分晋级和清空单场。
- 支持本地状态保存、导入导出、事件日志、撤销、完整性检查和自动修复。
- 支持静态部署，本机默认访问 `http://127.0.0.1:4176/`。

当前多人端目标是让裁判、队长和观众共享同一服务端权威状态：

- 裁判端拥有最高权限，可创建赛事、导入选手、制定顺序、暂停、恢复、回滚、修正和导出。
- 队长端只能在本人回合或本人海克斯窗口内操作，非本人回合只读观看当前回合队长视角。
- 观众端只读观看当前回合队长视角，不允许任何写操作。
- 服务端通过 command、event、projection 和 SSE 同步各端画面。
- 多人端支持 `6-20` 队可配置、双阵营/无阵营两种规则模式、替补激活与替换、多房间列表、归档和删除。
- 本地正式部署推荐 Docker Compose + PostgreSQL，使用 `HEXCORE_MAX_ROOMS` 控制 active 房间数量上限。

## 3. 核心规则

当前默认规则如下：

| 项目 | 当前值 |
| --- | --- |
| 队伍数量 | 裁判端维护版固定 10 队；多人端支持 6-20 队 |
| 每队人数 | 5 人，包含队长 |
| 阵营 | 双阵营：本地人 / 外地人；无阵营：公共卡池 |
| 抽选轮数 | 固定 4 轮 |
| 商店卡数 | 每次 5 张 |
| 费用档位 | 1-5 费，0 费为队长锁定展示 |
| 初始金币 | 6 |
| 轮收入 | 3 |
| 刷新费用 | 第 1-4 轮分别为 1/2/3/4 金币 |

选手分档优先使用 `resultScore`、`S1-S6` 赛季成绩和 `FMVP` 信息；缺少官方成绩时，按 `score` 直接落入同数值费用池。系统允许超过 50 名选手导入，超出组队需求的选手保持空闲。

## 4. 当前核心能力

HEXCORE 2.0 当前由以下模块提供核心能力：

- App State：规范化本地状态、兼容旧状态、维护设置和 UI 状态。
- Event Store：记录裁判操作、规则触发、失败原因和恢复动作。
- Turn Order Engine：计算四轮顺位、处理轮初提位、后移和顺位解释。
- Shop Engine：按阵营、费用池、概率和海克斯修饰生成商店。
- Economy Engine：处理金币、刷新费用、折扣、返还和收入。
- Hexcore Engine：执行金币模式海克斯、轮初效果、目标选择和响应窗口。
- Assignment Engine：处理购买入队、海克斯入队、随机补位和同阵营硬校验。
- Multiplayer Server：处理房间、session、command、projection、SSE、PostgreSQL 持久化和房间生命周期。
- Integrity Service：检查并修复跨阵营、重复归属、超员、禁用选手和卡池异常。
- Export Service：导出状态、日志和选手数据。

## 5. 部署与入口

本项目是静态前端应用，使用本地 Node 静态服务运行：

```bash
npm start
```

默认服务地址：

```text
http://127.0.0.1:4176/
```

`file://` 直接打开不受支持，页面会提示使用本地服务访问。静态服务只允许访问 `index.html`、`src/` 和 `public/assets/` 下的资源，并由回归测试覆盖路径穿越防护。

## 6. 长期演进目标

多人实时系统已经进入实现阶段，当前仍需继续完善：

- 无阵营海克斯文案和特殊分支继续扩展回归。
- PostgreSQL 正式部署演练和跨实例恢复验收。
- 客户现场演示手册。

多人端开发不得破坏裁判端稳定性；裁判端、队长端和观众端的权限边界必须由服务端校验，而不是只靠前端隐藏按钮。

## 7. 当前工作原则

- 裁判端是最高权限入口；队长端只在规则允许窗口内写入；观众端始终只读。
- 所有关键操作都必须产生事件日志。
- UI 不直接绕过业务引擎修改核心状态。
- 双阵营模式下入队、补位、海克斯和商店必须执行同阵营校验；无阵营模式下使用公共候选池，不能显示阵营误导文案。
- 替补和缺席信息由裁判端管理；队长端和观众端只能看到公开替换结果。
- 金币、购买权、刷新次数和响应窗口必须可解释、可回滚、可复盘。
- 文档、回归测试和浏览器验证需要随功能变更同步更新。
