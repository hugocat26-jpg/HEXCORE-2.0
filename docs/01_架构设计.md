# 架构设计

## 1. 当前阶段架构

当前阶段是裁判单端应用：

```text
裁判浏览器
  ├─ UI 页面
  ├─ 应用状态 AppState
  ├─ 规则引擎
  ├─ 事件日志
  └─ 本地存储
```

裁判执行所有操作。系统不向队长开放写权限。

## 2. 未来多人架构边界

当前模块边界按未来多人架构预留：

```text
客户端
  ├─ 裁判端
  ├─ 队长端
  ├─ 观众端
  └─ 大屏端

服务端
  ├─ Auth
  ├─ Permission
  ├─ Draft Engine
  ├─ Hexcore Engine
  ├─ Event Store
  ├─ Realtime Gateway
  └─ Database
```

第一阶段不实现服务端，但业务代码应避免依赖“前端直接可信”的假设。

## 4. 单向数据流

```text
裁判操作
  ↓
Action
  ↓
Engine 校验并计算结果
  ↓
生成 Domain Event
  ↓
更新 AppState
  ↓
保存 Event Log / Snapshot
  ↓
UI 重新渲染
```

禁止 UI 直接改 `players`、`captains`、`draft` 等核心状态。

## 5. 引擎职责

### Draft Engine

- 控制流程状态机。
- 判断当前阶段允许哪些操作。
- 处理开始轮抽、进入下一位、进入下一轮、完成轮抽。

### Turn Order Engine

- 生成基础蛇形顺位。
- 应用顺位类海克斯。
- 处理顺位冲突和优先级。

### Pool Engine

- 根据回合计算基础卡池。
- 应用个人卡池映射。
- 应用全局卡池交换。
- 输出当前有效卡池。

### Probability Engine

- 从有效卡池中计算抽卡候选。
- 应用权重、保底、排除和特殊概率。
- 生成可记录的随机结果。

### Hexcore Engine

- 读取海克斯定义。
- 识别触发时机。
- 调用对应 effect handler。
- 生成运行时效果。

### Visibility Engine

- 控制卡片身份是否可见。
- 支持致盲、暗牌、身份互换、延迟揭示。
- 当前裁判代执行阶段仍需要该模块，用于提前验证未来多人规则。

### Assignment Engine

- 添加队员。
- 自动随机分配。
- 处理转队。
- 处理补偿回合。
- 队伍满员后跳过后续选人。

## 6. 当前阶段权限边界

当前只开放一个可写角色：

```text
referee
```

未来角色预留：

```text
super_admin
tournament_admin
referee
supervisor
captain
viewer
```

当前阶段所有写操作都要求 `referee` 权限。队长端只作为后续视图扩展，不参与第一阶段实现。
