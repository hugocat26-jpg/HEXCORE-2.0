# 架构设计

更新时间：2026-05-25

当前状态：本文记录当前原生 JavaScript 静态应用架构，并保留未来多人实时系统的边界设想。

## 1. 当前阶段架构

当前阶段是裁判单端应用：

```text
裁判浏览器
  ├─ UI 页面
  ├─ 应用状态 AppState
  ├─ 规则引擎
  ├─ 事件日志
  └─ 本地存储
```

裁判执行所有操作。系统不向队长开放写权限。

## 2. 未来多人架构边界

当前模块边界按未来多人架构预留：

```text
客户端
  ├─ 裁判端
  ├─ 队长端
  ├─ 观众端
  └─ 大屏端

服务端
  ├─ Auth
  ├─ Permission
  ├─ Draft Engine
  ├─ Hexcore Engine
  ├─ Event Store
  ├─ Realtime Gateway
  └─ Database
```

第一阶段不实现服务端，但业务代码应避免依赖“前端直接可信”的假设。

## 4. 单向数据流

```text
裁判操作
  ↓
Action
  ↓
Engine 校验并计算结果
  ↓
生成 Domain Event
  ↓
更新 AppState
  ↓
保存 Event Log / Snapshot
  ↓
UI 重新渲染
```

禁止 UI 直接改 `players`、`captains`、`draft` 等核心状态。

## 5. 引擎职责

### Draft Engine

- 控制流程状态机。
- 判断当前阶段允许哪些操作。
- 处理开始金币商店抽选、进入下一位、进入下一轮、完成四轮抽选。

### Turn Order Engine

- 生成基础蛇形顺位。
- 应用顺位类海克斯。
- 处理顺位冲突和优先级。

### Pool Engine

- 按阵营、队长锁定、禁用状态和归属状态计算有效候选池。
- 按费用分档和官方成绩来源输出 1-5 费卡池。
- 保证队长不进入购买、补位或海克斯入队候选。

### Probability Engine

- 从有效卡池中按当前轮金币商店概率计算候选。
- 应用空费用等比放大、保底、排除和特殊概率。
- 生成可记录的随机结果。

### Hexcore Engine

- 读取海克斯定义。
- 识别触发时机。
- 调用对应 effect handler。
- 生成运行时效果。

### Visibility Engine

- 控制卡片身份是否可见。
- 支持致盲、暗牌、身份互换、延迟揭示。
- 当前裁判代执行阶段仍需要该模块，用于提前验证未来多人规则。

### Assignment Engine

- 处理购买入队、海克斯入队和最终随机补位。
- 执行同阵营、非队长、非禁用、容量和重复归属硬校验。
- 处理同阵营置换与海克斯补偿，禁止跨阵营置换。
- 队伍满员后跳过后续选人。

## 6. 当前阶段权限边界

当前只开放一个可写角色：

```text
referee
```

未来角色预留：

```text
super_admin
tournament_admin
referee
supervisor
captain
viewer
```

当前阶段所有写操作都要求 `referee` 权限。队长端只作为后续视图扩展，不参与第一阶段实现。
