# UI 设计规范

## 1. 当前阶段界面目标

当前阶段界面是裁判工作台。所有写操作由裁判执行，页面不提供队长登录、观众端或大屏端入口。

主界面必须支持：

- 查看当前阶段、当前轮次、当前队长、当前金币和当前任务。
- 导入、筛选、编辑选手，并设置 10 名队长。
- 查看本地人、外地人双阵营选手池和费用分档。
- 为队长执行海克斯五抽一、刷新候选和选择海克斯。
- 代队长生成金币商店、刷新商店、点击选手卡直接购买。
- 处理海克斯目标选择、轮初顺位效果、购买后响应窗口和入队揭示。
- 查看 10 队紧凑阵容看板、金币、海克斯、费用分布和异常状态。
- 生成阵营对抗赛程、拖拽首轮队伍、确认轮空、录入比分和清空单场。
- 查看事件日志、导出状态、执行完整性检查和本地数据修复。

## 2. 信息架构

应用采用固定裁判控制台结构：

```text
左侧导航
顶部状态栏
中间主工作区
右侧或底部事件日志
底部或主区内队伍阵容看板
```

当前主导航包括：

| 页面 | 用途 |
| --- | --- |
| 实时抽选 | 主持金币商店抽选、购买、刷新、跳过和海克斯执行。 |
| 队伍管理 | 查看和修正 10 队阵容、队长、阵营和异常。 |
| 选手库 | 导入、筛选、新增、禁用和编辑选手。 |
| 海克斯库 | 查看海克斯分类、说明、详情和启用状态。 |
| 赛程 | 生成阵营对抗框架、拖拽首轮、确认轮空和录入比分。 |
| 规则设置 | 调整金币、刷新、队伍容量、模板和海克斯禁用项。 |
| 日志 | 查询、定位、导出和清空事件日志。 |
| 系统设置 | 状态导入导出、完整性检查、自动修复和清理本地数据。 |

## 3. 视觉方向

视觉风格为冷静、紧凑、专业的赛事裁判控制台。

色彩：

- 背景：深炭黑。
- 主边框：冷灰蓝。
- 主要强调：青色，用于当前操作、可点击主动作和在线状态。
- 次强调：琥珀色，用于金币、当前队长、顺位变化和待确认状态。
- 成功状态：绿色。
- 危险状态：红色。
- 阵营标签：本地人和外地人使用可区分但克制的标签色，不主导页面。

避免：

- 营销页 hero。
- 大面积装饰背景。
- 紫色主导渐变。
- 卡片套卡片。
- 过度圆角和厚重阴影。
- 用大段说明文字占据工作区。

## 4. 顶部状态栏

顶部状态栏必须提供裁判快速判断所需的最小信息：

- `HEXCORE 2.0`。
- 裁判代执行状态。
- 当前页面或流程阶段。
- 当前轮次。
- 当前队长。
- 当前任务。
- 本地保存或运行状态。
- 常用快捷动作，例如刷新、导出或系统检查入口。

状态栏文字必须可在窄宽下换行或省略，不允许遮挡按钮。

## 5. 实时抽选页

实时抽选页是最高优先级工作面。

必须突出：

- 当前队长和当前轮次。
- 当前顺位及顺位变化原因。
- 当前商店 5 张卡。
- 当前队长金币、免费刷新次数、购买权状态。
- 可执行海克斯、受阻原因和响应窗口。
- 主操作按钮：开始抽卡、生成商店、刷新商店、跳过、下一位。

选手卡交互：

| 状态 | 规则 |
| --- | --- |
| 可购买 | hover/focus 时高亮并显示点击购买语义。 |
| 点击 | 直接尝试购买该卡，不再需要二段式“购买此卡”按钮。 |
| 不可购买 | 点击后给出明确原因，例如金币不足、队伍已满、购买权已用、卡槽为空。 |
| 已购买 | 卡片弱化并保留日志可追溯信息。 |
| 信息打乱 | 显示身份和真实身份按规则处理，购买后揭示真实选手。 |

## 6. 队伍阵容看板

实时抽选页应提供 10 队紧凑阵容看板，裁判不离开主流程即可判断全局状态。

每张队伍卡至少显示：

- 队伍序号和队名。
- 阵营。
- 人数，例如 `5/5`。
- 队长。
- 剩余金币。
- 持有海克斯摘要。
- 队员摘要。
- 费用分布。
- 当前状态，例如当前操作、已满员、待补位、已购买、已跳过、异常。

交互规则：

- 当前操作队伍必须明显高亮。
- 异常队伍必须有红色状态点和明确原因。
- hover 详情显示完整阵容，不遮挡主要购买按钮。
- 点击队伍卡只定位到队伍管理页，不改变抽选状态。
- 文本过长时省略，详情层展示完整文本。

## 7. 海克斯界面

海克斯卡片必须使用统一结构：

- PNG 图标优先，SVG 图标兜底。
- 名称。
- 主分类。
- 触发方式。
- 使用次数或被动说明。
- 效果摘要。
- 详情入口。

详情交互：

- 点击详情打开统一弹窗。
- 弹窗包含规则介绍、使用技巧、注意事项和日志口径。
- 弹窗固定最大宽高，长内容在内部滚动。
- 右上角 X、遮罩点击和 Esc 都能关闭。
- 打开或关闭详情不得选择、分配或执行海克斯。

图标规范：

- 当前金币模式海克斯使用 `public/assets/hex-icons/<hexcore-id>.png`。
- 图标透明背景，主体居中，少细节，`28x28` 下仍可辨认。
- 缺失 PNG 时必须回退 SVG，不出现空白。

## 8. 选手库与导入

选手库必须清晰区分：

- 本地人 / 外地人。
- 队长锁定 / 可抽 / 已入队 / 禁用。
- 1-5 费费用档位。
- 官方成绩分档和 `score` 直落分档来源。

导入预览必须显示：

- 新增数。
- 跳过数。
- 重复 ID。
- 缺字段。
- 非法评分。
- 阵营缺失或无法识别。
- 可导入选手的勾选状态。

导入人数不再按 50 人截断；超过组队需求的选手应作为空闲候选保留。

## 9. 赛程页

赛程页服务于选人完成后的阵营对抗框架。

必须支持：

- 按阵营生成首轮框架。
- 裁判拖拽队伍进入首轮槽位。
- 单边队伍进入待补齐状态，不自动轮空。
- 裁判手动确认轮空。
- 已确认轮空后可移出队伍。
- 清空单场会清空该场 A/B 队伍、比分、胜者和轮空确认。
- 保存比分后自动推进后续晋级链。

赛程操作不得改变选人队伍成员、金币、海克斯或抽选流程状态。

## 10. 事件日志

事件日志按时间倒序展示，必须覆盖：

- 队伍和选手导入。
- 队长设置。
- 海克斯抽取、刷新、选择、分配和执行。
- 商店生成、刷新、购买、失败原因。
- 金币变化和返还。
- 顺位变化。
- 响应窗口、入队揭示和随机补位。
- 赛程生成、拖拽、轮空、比分和清空。
- 完整性检查和自动修复。

日志文本必须可定位、可导出、可过滤；用户输入内容必须转义后展示。

## 11. 响应式要求

桌面优先，兼顾窄宽窗口。

要求：

- 主商店和主操作按钮优先保持可见。
- 左侧导航在窄宽下可压缩或转为顶部导航。
- 右侧日志可移到底部。
- 商店卡从多列降为单列或两列时不重叠。
- 队伍看板可换行或横向滚动。
- 弹窗不越出视口，正文内部滚动。
- 按钮文字不挤出容器，必要时换行或缩短标签。

## 12. 禁止项

- 不在 UI 层绕过引擎直接改队伍、金币或选手归属。
- 不用纯颜色表达唯一状态，必须同时有文字或图标。
- 不让 hover 承载唯一关键信息；可点击详情必须可稳定阅读。
- 不在抽选主流程中放置长篇教程文案。
- 不允许商店、海克斯、补位或赛程操作造成跨阵营入队。
- 不允许缺失图标、长名称或窗口缩放导致布局坍塌。
