# 四轮金币抽卡模式实施修改方案

归档状态：历史实施方案，当前四轮金币商店模式和阵营锁定规则已完成落地。本文保留迁移过程、风险拆分和当时的执行路径，不作为当前待办计划；现行维护以 `06_开发计划.md` 和 `14_阵营锁定10队金币商店模式_实施规范.md` 为准。

## 1. 目标

本文记录 HEXCORE 2.0 从早期无金币轮抽模型切换为“四轮金币抽卡模式”的历史实施路径。切换后的系统默认服务于 50 人参赛、10 队组队、每队 1 名队长加 4 名队员的赛事流程。

本方案覆盖从参赛数据准备、裁判指定队长、选手费用分档、金币经济、商店刷新购买、最终补位、旧存档迁移、UI 调整、测试验证、任务后置钩子、安全审查到 Gitee 上传的完整实施路径。

## 2. 已确认规则口径

### 2.1 参赛与队伍

- 总参赛人数固定为 50 人。
- 队长由裁判从 50 名参赛选手中指定。
- 系统默认队伍数量改为 10 队。
- 每队最终阵容为 1 名队长加 4 名队员。
- 队长进入队长专属池，不进入队员抽卡卡池。
- 剩余 40 名非队长选手进入队员卡池。

### 2.2 金币与购买

- 金币归队长独立拥有。
- 裁判只代为执行金币消费、刷新、购买、跳过等操作。
- 每名队长开局 6 金币。
- 第一轮不发放回合收入。
- 第二、三、四轮开始时，每名队长各获得 3 金币。
- 未花费金币保留至下一轮。
- 无利息、无额外金币奖励、无隐藏收益。
- 每名队长每轮首次进入操作时，系统免费生成一次 5 张商店。
- 之后每次刷新扣金币：1、2、3、4，第四次及以后固定 4 金币。
- 每轮每名队长最多购买 1 名队员。
- 跳过本轮后，本轮购买权限立即作废，不能通过裁判跳转重新购买。

### 2.3 费用与卡池

- 历届 FMVP 若参加本届且不是队长，直接定为 5 费卡。
- 历届 FMVP 若未参加，或作为队长参赛，则不进入队员卡池。
- 同一选手多次 FMVP 仍只生成 1 张 5 费卡。
- 当前比赛只举办过 6 届，本届为第 7 届，因此 5 费卡最多 6 张，最少 0 张。
- 非 FMVP 选手按过去 6 届成绩累计评分。
- 非 FMVP 分档继续使用比例：4 费前 15%，3 费接下来 25%，2 费接下来 35%，1 费剩余。
- 同分排序：冠军次数、亚军次数、4 强次数、参赛次数、最近一届成绩、随机。

### 2.4 最终补位

- 四轮结束后，若某队不足 4 名队员，则从所有未被购买的剩余队员中随机补位。
- 补位不区分费用，1-5 费均可能被随机分配。
- 补位不消耗金币，不允许选择，不允许刷新。
- 补位结果写入日志，并作为最终阵容固化。

### 2.5 海克斯

- 队员抽卡阶段暂时禁用所有入队型海克斯。
- 禁用范围包括额外入队、免费入队、转队、补偿回合、额外选人、自动分配、直接自选队员。
- 无金币模式删除，全面切换为金币模式。
- 旧存档仅作为历史备份，不迁移旧抽卡结果和旧无金币轮次状态。

## 3. 当前系统影响面

当前系统主要由静态前端和本地状态组成，核心文件分布如下。

| 模块 | 当前职责 | 金币模式影响 |
| --- | --- | --- |
| `src/core/app-state.js` | 状态默认值、状态规范化、选择器 | 需要加入 10 队默认、5 费、FMVP、队长经济状态、旧存档迁移 |
| `src/engines/probability-engine.js` | 当前按单一 tier 抽候选卡 | 需要改为按轮次概率生成 5 张商店 |
| `src/engines/pool-engine.js` | 计算当前有效卡池 tier | 需要支持 1-5 费概率池，不再只按单一 tier |
| `src/engines/assignment-engine.js` | 入队、随机分配、转队 | 购买入队需要扣金币；最终补位需要独立来源 |
| `src/engines/hexcore-engine.js` | 海克斯执行和冲突处理 | 金币模式禁用入队型海克斯，保留状态展示 |
| `src/main.js` | 裁判动作入口 | 需要新增开店、刷新、购买、跳过固化、最终补位动作 |
| `src/ui/referee-console.js` | 全部 UI 渲染 | 需要展示金币、商店、刷新费用、购买状态、概率规则、补位状态 |
| `src/services/export-service.js` | 状态导入导出和选手导入 | 需要支持 FMVP/历史成绩字段导入和旧存档迁移提示 |
| `scripts/regression.js` | 回归测试 | 需要新增金币模式、商店、购买、跳过、补位、迁移测试 |

### 3.1 当前功能遍历与适配结论

| 系统功能 | 旧逻辑 | 新适配结果 |
| --- | --- | --- |
| 默认参赛数据 | 示例队伍多于10队，且存在已入队队员 | 默认载入50名参赛选手、10名队长、40名队员卡池，队伍从空队员位开始 |
| 队长确认 | 队长可从选手中绑定 | 保留裁判指定队长；被指定队长进入队长专属池，不进入队员商店 |
| 费用分档 | 非队长按评分四等分为1-4档 | 参赛且非队长的历届FMVP直接为5费；其余非FMVP按历届成绩积分和比例分为1-4费 |
| 抽卡入口 | 按单一tier抽若干张候选卡 | 改为每轮首次免费生成5张商店，后续刷新扣金币 |
| 刷新逻辑 | 刷新不消耗资源 | 每名队长每轮刷新费用独立递增：1、2、3、4，之后4金币封顶 |
| 购买逻辑 | 选择卡后直接入队 | 购买时校验金币、队伍容量、购买权限；扣除等于费用的金币后入队 |
| 跳过逻辑 | 跳过后仅前进顺位 | 跳过会立即写入本轮`skipped=true`，购买权限作废，跳转回来也不能购买 |
| 下一位逻辑 | 可直接切换队长 | 若当前队长未购买且未跳过，切换下一位时自动固化为跳过 |
| 裁判跳转 | 可跳到任意轮次和队长 | 金币模式只允许在当前轮次内定位队长，禁止手动跨轮跳转提前打开高费概率 |
| 后台补录 | 队伍页可手动补录队员 | 金币模式禁用手动补录；入队引擎只接受金币购买和终局随机补位两个来源 |
| 阵容移回 | 队伍页可将队员移回池 | 金币模式禁用手动移回，购买或补位后的阵容只能通过撤销或重置流程修正 |
| 配置变更 | 抽卡中仍可改队伍、顺位、规则和选手状态 | 开店、购买、跳过、进入后续轮次或已有队员后，基础配置和阵容操作固化 |
| 轮次推进 | 只推进轮次和顺位 | 第2-4轮开始时给所有队长发放3金币；四轮结束后触发随机补位 |
| 最终补位 | 无强制补位 | 阵容不足4名队员时，从剩余1-5费队员中随机补位，不消耗金币 |
| 入队型海克斯 | 可额外入队、转队、补偿回合、自选 | 金币模式下禁用会破坏金币购买权的海克斯，UI显示“金币模式禁用”；残留被动契约也不会触发免费入队 |
| 日志与复盘 | 记录抽卡和入队 | 记录免费商店、付费刷新、金币购买、跳过作废、最终补位和剩余金币 |
| 导入选手 | 支持基础字段 | 继续支持CSV/JSON，并兼容S1-S6成绩、FMVP届数字段；导入和新增均受50人总参赛上限约束 |
| 旧存档 | 直接规范化旧状态 | 非金币旧状态保存为`legacyNoGoldBackup`，清空旧抽卡结果和旧阵容，重建金币模式初始流程 |
| 回归测试 | 覆盖无金币抽卡和旧海克斯 | 改为覆盖默认配置、商店概率、刷新扣费、购买权限、跳过锁死、补位、导入、迁移和安全硬化 |

## 5. 新增或重构引擎

### 5.1 `economy-engine`

建议新增 `src/engines/economy-engine.js`，集中处理金币规则。

职责：

- 初始化队长金币。
- 在队长进入第 2-4 轮时发放 3 金币。
- 判断本轮收入是否已结算。
- 计算下一次刷新费用。
- 扣除刷新费用。
- 校验购买金币是否足够。
- 扣除购买费用。
- 标记购买权已使用。
- 标记跳过状态。

核心接口建议：

```js
ensureRoundIncome(captainId, round)
nextRefreshCost(captainId, round)
canRefresh(captainId, round)
payRefresh(captainId, round)
canPurchase(captainId, playerId)
payPurchase(captainId, playerId)
markSkipped(captainId, round)
roundState(captainId, round)
```

### 5.2 `shop-engine`

建议新增 `src/engines/shop-engine.js`，不要继续让 `probability-engine` 承担全部商店逻辑。

职责：

- 按轮次概率生成 5 张商店。
- 按费用档位抽取未购买、非队长、非禁用队员。
- 同一次商店不重复。
- 某档位空池时概率重分配。
- 免费初始商店和付费刷新统一走同一生成逻辑。

核心接口建议：

```js
roundProbabilities(round)
availablePlayersByTier()
normalizeProbabilities(probabilities, availableByTier)
drawTier(normalizedProbabilities)
generateShop(captainId, round, reason)
```

目标：让系统状态具备金币模式所需字段，并能兼容旧存档。

### 阶段 2：FMVP与历史成绩分档

目标：实现官方费用分档。

任务：

- 实现历史成绩分值：未参赛 0、1轮游 1、4强 3、亚军 5、冠军 7。
- FMVP 参赛且非队长时硬锁 5 费。
- 队长即使是 FMVP，也进入队长专属池，不进入 5 费池。
- 同一选手多次 FMVP 只生成一张 5 费卡。
- 5 费卡上限为 6。
- 非 FMVP 按 15%、25%、35%、剩余比例分配 4、3、2、1 费。
- 同分排序按冠军次数、亚军次数、4强次数、参赛次数、最近一届成绩、随机。

涉及文件：

- `src/core/app-state.js`
- `src/services/export-service.js`
- `src/ui/referee-console.js`
- `scripts/regression.js`

验收：

- 50 人数据导入后能稳定分档。
- 5 费数量不超过 6。
- 队长不会出现在队员卡池。
- 选手库页面能显示分档原因。

### 阶段 3：金币经济引擎

目标：实现队长独立金币所有权。

任务：

- 新增 `economy-engine.js`。
- 第 1 轮不发收入。
- 第 2-4 轮每名队长首次进入时发 3 金币。
- 金币保留到下一轮。
- 刷新费用每队长每轮独立计数并重置。
- 购买和刷新禁止导致金币为负。

涉及文件：

- `src/engines/economy-engine.js`
- `index.html`
- `scripts/regression.js`

验收：

- 每名队长开局 6 金币。
- 第 2、3、4 轮收入只发一次。
- 连续刷新费用为 1、2、3、4、4。
- 金币不足时不能刷新或购买。

### 阶段 4：商店与概率卡池

目标：替换无金币抽卡为 5 张商店。

任务：

- 新增 `shop-engine.js`。
- 每名队长每轮首次进入时免费生成 5 张商店。
- 付费刷新重新生成 5 张商店。
- 按官方概率表抽取费用档位。
- 空档位概率按剩余可用档位重分配。
- 同一次商店不重复展示同一选手。
- 已购买队员永久移出卡池。

涉及文件：

- `src/engines/shop-engine.js`
- `src/engines/probability-engine.js`
- `src/main.js`
- `scripts/regression.js`

验收：

- 第一轮不会出现 4 费、5 费。
- 第二轮可能出现 4 费，不出现 5 费。
- 第三轮有 2% 5 费概率。
- 第四轮 4 费、5 费概率最高。
- 空池时不会崩溃，概率能重分配。

- UI 明确显示“金币模式禁用”。
- 禁用海克斯不能被裁判触发。
- 保留未来扩展入口，但第一版不启用。

涉及文件：

- `src/core/sample-data.js`
- `src/engines/hexcore-engine.js`
- `src/ui/referee-console.js`

## 9. 风险与约束

### 9.1 旧海克斯冲突

现有多个海克斯会直接入队、免费分配、转队或补偿回合。它们与金币模式的“每轮一次购买、金币决定购买能力”冲突。第一版必须禁用。

### 9.2 旧存档不兼容

旧存档中的无金币抽卡状态无法可靠迁移到金币模式。迁移策略必须只保留选手、队伍、队长和历史信息，重建抽卡状态。

### 9.3 概率测试不可依赖单次随机

概率相关测试应测试结构约束和大量样本区间，不能要求单次抽卡命中特定费用。

### 9.4 最终补位可能随机到高费

这是已确认规则。UI 和日志需要明确提示：跳过或金币运营失败仍可能获得随机高费补位，但补位不可控。

## 10. 完成定义

本计划完成需要同时满足：

- 默认 10 队、50 人参赛模型可用。
- 队长从 50 人中指定，并进入队长专属池。
- 非队长选手按 FMVP 与历史成绩进入 1-5 费。
- 每名队长独立金币、收入、刷新费用和购买权正确。
- 四轮商店、刷新、购买、跳过流程完整。
- 四轮结束后自动随机补位并固化最终阵容。
- 入队型海克斯在金币模式中不可执行。
- 旧无金币模式不再作为默认玩法。
- 回归测试通过。
- UI 浏览器验证通过。
- Codex Security 全量审查完成，确认问题已修复或无问题。
