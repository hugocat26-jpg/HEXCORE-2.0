# HEXCORE 2.0 项目总览

## 1. 项目定位

HEXCORE 2.0 是海克斯选人抽卡系统的新架构版本。

术语边界：当前系统处理的是“队长通过抽卡和海克斯完成选人”的流程，不是比赛过程本身。后续可以增加对局排程或赛程安排模块，但抽卡选人流程应统一称为“选人流程”“抽选流程”或“轮抽流程”。

当前阶段采用“裁判代执行所有操作”的工作模式：

- 裁判登录系统。
- 裁判导入选手、设置队长、分配海克斯。
- 裁判主持轮抽流程。
- 裁判代替队长点击抽卡、使用海克斯、选择选手。
- 队长、观众、大屏只作为未来扩展目标，不在当前阶段开放操作权限。

系统设计仍预留多人协作边界，但第一版不实现队长登录自行抽卡。

## 2. 当前阶段目标

第一阶段目标是建立一个更可靠的单端裁判工作台：

- 取消金币买卡逻辑。
- 使用 `tier` 表示选手卡池档位。
- 建立轮抽状态机。
- 建立海克斯自动化引擎。
- 建立事件日志，记录所有关键操作。
- 支持数据导入、导出、回放、撤销的基础能力。
- 保持静态部署或本地运行的低成本。

## 3. 长期演进目标

长期目标是升级为多人实时系统：

- 队长登录自行抽卡。
- 裁判只负责主持、暂停、仲裁、回退。
- 裁判监督可审批高风险操作。
- 观众和大屏实时同步公开状态。
- 服务端成为唯一规则权威。
- 可扩展对局排程，但对局排程与抽卡选人流程保持模块边界。

因此当前版本的模块边界应按未来服务端迁移设计，但实现上先保持裁判单端操作。

## 4. 不再保留的旧业务

以下旧逻辑不进入 HEXCORE 2.0：

- 金币。
- 购买费用。
- 扣金币。
- 返金币。
- 破产。
- 金币补偿。
- 金币编辑。
- 金币不足限制。

选手的档位仅代表抽卡池：

| tier | 名称 |
| --- | --- |
| 1 | 侏儒马 |
| 2 | 中等马 |
| 3 | 上等马 |
| 4 | 猛犸 |

## 5. 核心能力

HEXCORE 2.0 的核心能力由以下引擎提供：

- Draft Engine：控制轮抽流程是否合法。
- Turn Order Engine：计算每轮最终顺位。
- Pool Engine：计算队长当前有效卡池。
- Probability Engine：控制抽卡权重、概率和随机结果。
- Hexcore Engine：执行海克斯效果。
- Visibility Engine：控制暗牌、致盲、延迟揭示。
- Assignment Engine：处理入队、转队、补偿回合和自动分配。

## 6. 当前工作原则

- 裁判端是唯一可写入口。
- 所有操作都必须产生事件日志。
- UI 不直接修改业务数据。
- 引擎不直接操作 DOM。
- 海克斯尽量用规则原语配置，不写散乱特殊分支。
- 当前版本可本地保存，后续可替换为服务端存储。
