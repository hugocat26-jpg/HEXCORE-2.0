# HEXCORE 2.0 项目总览

## 1. 项目定位

HEXCORE 2.0 是面向线下赛事的裁判单端选人抽卡控制台。当前运行版本采用“阵营锁定 10 队金币商店模式”：裁判导入选手、设置队长、分配海克斯、主持四轮金币商店抽选，并在需要时修正队伍、赛程和本地状态。

术语边界：

- 本系统处理“队长通过金币商店和海克斯完成选人”的流程，不处理实际对局内操作。
- 当前系统内的“比赛”“赛程”只指选人结束后的阵营对抗框架和比分记录。
- 队长、观众、大屏和多人同步仍是后续扩展目标；当前版本只有裁判端拥有写权限。

## 2. 当前阶段目标

当前阶段目标是交付可本地运行、可审计、可回归验证的裁判工作台：

- 固定 10 队，每队 5 人，队长占用 1 个名额。
- 支持本地人、外地人双阵营，队长和队员必须同阵营。
- 支持 1-5 费金币商店，四轮抽选，每次商店 5 张卡。
- 支持初始金币、轮收入、刷新费用、折扣、返还和海克斯经济效果。
- 支持海克斯五抽一、PNG 图标、详情弹窗、执行队列和响应窗口。
- 支持赛程生成、首轮拖拽、确认轮空、比分晋级和清空单场。
- 支持本地状态保存、导入导出、事件日志、撤销、完整性检查和自动修复。
- 支持静态部署，本机默认访问 `http://127.0.0.1:4176/`。

## 3. 核心规则

当前默认规则如下：

| 项目 | 当前值 |
| --- | --- |
| 队伍数量 | 固定 10 队 |
| 每队人数 | 5 人，包含队长 |
| 阵营 | 本地人 / 外地人 |
| 抽选轮数 | 固定 4 轮 |
| 商店卡数 | 每次 5 张 |
| 费用档位 | 1-5 费，0 费为队长锁定展示 |
| 初始金币 | 6 |
| 轮收入 | 3 |
| 刷新费用 | 第 1-4 轮分别为 1/2/3/4 金币 |

选手分档优先使用 `resultScore`、`S1-S6` 赛季成绩和 `FMVP` 信息；缺少官方成绩时，按 `score` 直接落入同数值费用池。系统允许超过 50 名选手导入，超出组队需求的选手保持空闲。

## 4. 当前核心能力

HEXCORE 2.0 当前由以下模块提供核心能力：

- App State：规范化本地状态、兼容旧状态、维护设置和 UI 状态。
- Event Store：记录裁判操作、规则触发、失败原因和恢复动作。
- Turn Order Engine：计算四轮顺位、处理轮初提位、后移和顺位解释。
- Shop Engine：按阵营、费用池、概率和海克斯修饰生成商店。
- Economy Engine：处理金币、刷新费用、折扣、返还和收入。
- Hexcore Engine：执行金币模式海克斯、轮初效果、目标选择和响应窗口。
- Assignment Engine：处理购买入队、海克斯入队、随机补位和同阵营硬校验。
- Integrity Service：检查并修复跨阵营、重复归属、超员、禁用选手和卡池异常。
- Export Service：导出状态、日志和选手数据。

## 5. 部署与入口

本项目是静态前端应用，使用本地 Node 静态服务运行：

```bash
npm start
```

默认服务地址：

```text
http://127.0.0.1:4176/
```

`file://` 直接打开不受支持，页面会提示使用本地服务访问。静态服务只允许访问 `index.html`、`src/` 和 `public/assets/` 下的资源，并由回归测试覆盖路径穿越防护。

## 6. 长期演进目标

后续可演进为多人实时系统：

- 队长登录自行抽卡、刷新和确认购买。
- 裁判负责主持、暂停、仲裁、回退和高风险审批。
- 观众端和大屏端实时同步公开状态。
- 服务端成为唯一规则权威，前端只提交意图和展示结果。
- 赛程模块扩展为完整对局编排，但仍与选人抽卡状态机保持边界。

当前版本的模块边界应继续服务于未来迁移，但实现上优先保证裁判单端稳定。

## 7. 当前工作原则

- 裁判端是唯一可写入口。
- 所有关键操作都必须产生事件日志。
- UI 不直接绕过业务引擎修改核心状态。
- 入队、补位、海克斯和商店都必须执行同阵营校验。
- 金币、购买权、刷新次数和响应窗口必须可解释、可回滚、可复盘。
- 文档、回归测试和浏览器验证需要随功能变更同步更新。
