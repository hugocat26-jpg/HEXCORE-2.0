## 当前产品口径

HEXCORE 2.0 当前维护方向是“裁判端最高权限 + 队长端按回合操作 + 观众端只读观看当前回合队长视角”的抽卡选队友系统。多人端以服务端权威状态为准，通过 command、event、projection 和 SSE 同步裁判端、队长端和观众端。

当前不建设大屏端。

## M12 可配置队伍、多规则模式与本地 PostgreSQL 多房间管理

状态：已完成，标准验收通过，作为 M13 交付验证的前置能力。

### 目标

把当前固定 10 队、双阵营、单演示房间口径升级为：6-20 队可配置、支持双阵营/无阵营、替补可管理、多房间可管理，并以 Docker PostgreSQL 作为本地正式部署方案。

### 当前执行结果

- 多人端已完成 6-20 队创建、双阵营/无阵营规则、公共卡池、替补激活与替换、多房间列表、归档、删除和 `HEXCORE_MAX_ROOMS` active 房间上限。
- 无阵营 `海浪，我没吃饭` 和 `神兵天降` 已进入服务端权威结算；队长端通过 command/SSE 同步，不依赖本地伪造结果。
- 无阵营赛程默认全随机，并隐藏阵营对抗、班德尔保卫战、本地/外地槽位和阵营 A/B 等误导文案。
- 多人端裁判投影已同步动态 `teamCount`、`campMode`、`pairingMode` 和服务端 `team-*` 队伍 ID，避免新建 12 队房间进入裁判端后回落到本地默认 10 队。
- Docker Compose + PostgreSQL `.env.example`、schema、store 和部署运维说明已补齐。
- 当前回归：`npm test` 通过，输出 `regression ok: 60 tests`；无头 Chrome 已完成多人端创建无阵营 12 队、进入裁判端和赛程页验收，未发现相关控制台异常。
- M12 版本已同步为 `v2.0.19`，`CHANGELOG.md` 已记录 M12 多规则多人端、多房间、PostgreSQL/Docker 与投影同步收口。

### 已确认产品决策

- 队伍数量范围：`6-20` 队。
- 双阵营模式：队伍数必须为偶数，本地/外地队伍数量相等。
- 无阵营模式：不要求选手或队长阵营，所有队伍共用公共卡池。
- 无阵营海克斯：不整体禁用，采用“改写保留”。所有同阵营/本阵营筛选改为公共卡池；`海浪，我没吃饭` 和 `神兵天降` 使用无阵营改写版。
- 多房间管理：支持房间列表、归档、删除和最大房间数配置。
- 本地正式部署：使用 Docker Compose 启动 `hexcore + PostgreSQL`。

### 实施拆分

#### 1. 规则配置与状态归一化

目标：先解除固定 10 队、固定双阵营和固定阵营校验，否则后续 UI/API 都会被旧状态重新压回 10 队。

需关联：

- `apps/multiplayer/src/core/app-state.js`
- `packages/shared/index.js`
- `packages/rules/index.js`
- `apps/server/memory-store.js`
- `apps/server/projections.js`
- `scripts/regression.js`

新增或统一以下规则字段：

| 字段 | 说明 |
| --- | --- |
| `settings.teamCount` | 当前赛事队伍数量，范围 `6-20`；兼容旧 `totalTeams`。 |
| `settings.playersPerTeam` | 每队人数，范围 `2-8`，默认含队长。 |
| `settings.campMode` | `dual_camp` 双阵营或 `no_camp` 无阵营；旧状态默认 `dual_camp`。 |
| `settings.pairingMode` | `camp_versus`、`random` 或 `manual`；无阵营默认 `random`。 |
| `settings.allowSubstitutes` | 是否启用替补池，默认启用。 |
| `HEXCORE_MAX_ROOMS` | 服务端允许的最大 `active` 房间数。 |

执行要求：

- 默认仍创建 10 队，旧导出、旧本地缓存、旧房间状态导入后保持可运行。
- `normalizeState` 不再强制 10 队；`workflowChecklist`、队长确认、海克斯抽取顺序、金币商店顺位、赛程入口全部按 `teamCount` 计算。
- 双阵营模式下 `teamCount` 必须为偶数，并校验队长锁定选手阵营为 `teamCount / 2` 对 `teamCount / 2`。
- 无阵营模式下允许选手 `camp` 为空；旧有带阵营数据仍可保留但不参与筛选。
- 规则修改只允许在数据准备阶段进行；进入海克斯抽取后禁止修改 `teamCount`、`campMode`、`playersPerTeam`，除非裁判先重置流程。

需关联：

- `apps/multiplayer/src/ui/referee-console.js`
- `apps/multiplayer/src/main.js`
- `apps/server/memory-store.js`
- `apps/server/server.js`
- `apps/server/projections.js`

执行要求：

- 裁判端创建赛事、规则设置页、队伍管理页统一使用 `teamCount`，不再出现固定 10 队文案。
- 增加队伍时补齐队长、队长码、经济状态、海克斯抽取状态、顺位状态和赛程占位。
- 减少队伍时若被删除队伍已有队长锁定、普通队员、海克斯、金币商店记录、赛程记录或 session 绑定，必须要求裁判先重置相关流程。
- 队长端和观众端只展示当前房间实际队伍数，不显示不存在的队伍入口。
- 房间码生成必须随队伍数量变化，不能继续固定生成 10 个队长码。

#### 3. 双阵营与无阵营卡池规则

目标：把“可抽选手池”抽象成规则模式感知能力，避免商店、购买、海克斯、补位各自写阵营判断。

需关联：

- `apps/multiplayer/src/core/app-state.js`
- `apps/multiplayer/src/engines/shop-engine.js`
- `apps/multiplayer/src/engines/assignment-engine.js`
- `apps/multiplayer/src/engines/economy-engine.js`
- `apps/multiplayer/src/services/integrity-service.js`
- `apps/server/shop-service.js`
- `packages/rules/index.js`

执行要求：

- 新增或改造统一选择器，例如 `isCampMode()`、`eligiblePlayersForCaptain(captainId, excludedIds)`、`canAssignPlayerToCaptain(captainId, playerId)`。
- 双阵营模式：保持当前同阵营抽卡、同阵营购买、同阵营补位和跨阵营拒绝。
- 无阵营模式：商店从公共未占用选手池抽卡，购买和入队不校验阵营，最终补位也从公共池补齐。
- 所有拒绝提示按模式变化：双阵营显示“不能购买异阵营选手”，无阵营不得出现阵营相关拒绝语。
- 导入和完整性修复在双阵营模式下继续清理非法跨阵营入队；无阵营模式下不得把无 `camp` 选手误判为非法。

#### 4. 无阵营海克斯适配

目标：无阵营模式下保留海克斯玩法，但所有规则说明、按钮状态、日志和执行结果都不能继续依赖阵营概念。

需关联：

- `apps/multiplayer/src/core/sample-data.js`
- `apps/multiplayer/src/engines/hexcore-engine.js`
- `apps/multiplayer/src/engines/shop-engine.js`
- `apps/multiplayer/src/ui/referee-console.js`
- `packages/rules/index.js`
- `scripts/generate-hexcore-pdfs.py`

分类口径：

| 类型 | 海克斯 | 无阵营处理 |
| --- | --- | --- |
| 原样保留 | 压价券、抬价干扰、捐赠、赞助回流、开饭啦、吸血习性、巨人杀手、摄影艺术家、贤者的博爱、启元、大炮已充能、骤雨 血雾 清风、雪定饿的喵 | 逻辑不变，文案去掉阵营误导。 |
| 公共池改写 | 阵营侦察、保留席位、加急调货、稳健补强、舞会女王、神秘盲盒、黄金阶、棱彩阶、知识来源于分解、和我困在一起、背水一战 | 同阵营/本阵营候选改为公共可选池。 |
| 特殊改写 | 海浪，我没吃饭 | 仍然金币清零、跳过本轮、随机命中其他队长购买；命中后不再判断同/异阵营，发动者未满则直接夺取，原购买队长返还金币、购买权和补偿；取消“异阵营退回后轮末同阵营补偿”分支。 |
| 特殊改写 | 神兵天降 | 购买成功后响应窗口内可夺取任意其他队长刚购买的选手；仍限制每局 1 次，成功后发动者跳过下一轮，原购买队长返还金币和购买权。 |

执行要求：

- 海克斯详情弹窗根据 `campMode` 显示对应规则说明。
- 图录筛选、按钮状态、失败原因和事件日志按模式显示，不能在无阵营模式下出现“同阵营、本阵营、异阵营、阵营锁定”。
- `海浪`、`神兵天降` 的服务端命令校验、SSE 事件和本地裁判端执行结果必须一致。
- 旧双阵营回归测试必须继续覆盖同阵营/异阵营分支，新增无阵营测试覆盖改写分支。

#### 6. 替补池与替换流程

目标：把已存在的 `attendanceStatus = substitute` 从抽卡权重过滤扩展成裁判可操作流程。

需关联：

- `apps/multiplayer/src/core/app-state.js`
- `apps/multiplayer/src/ui/referee-console.js`
- `apps/multiplayer/src/services/export-service.js`
- `apps/server/projections.js`
- `packages/rules/index.js`
- `scripts/regression.js`

执行要求：

- 选手库区分正式池、替补池、缺席/高风险筛选。
- 替补默认不进入普通商店池；裁判激活后才进入正式候选池。
- 裁判可把缺席选手替换为替补；已入队缺席选手被替换时，原队伍成员、选手状态、事件日志和审计日志同步更新。
- 双阵营模式下替补必须匹配目标队伍阵营；无阵营模式下替补不受阵营限制。
- 队长端和观众端只看到公开替换结果，不显示缺席原因、裁判备注、风险等级和档案内部说明。

#### 7. 多房间生命周期管理

目标：从“能创建多个房间”升级为“能安全管理多个房间”，并明确不同房间之间不会混淆数据。

需关联：

- `apps/server/memory-store.js`
- `apps/server/postgres-store.js`
- `apps/server/postgres/schema.sql`
- `apps/server/server.js`
- `apps/server/projections.js`
- `apps/multiplayer/src/ui/referee-console.js`

执行要求：

- 房间状态支持 `active`、`archived`、`deleted`。
- 新增房间列表能力，显示赛事 ID、名称、创建时间、状态、队伍数、规则模式、在线订阅数、存储类型和最近更新时间。
- 创建房间前校验赛事 ID 重复、active 房间数量上限、队伍数量、规则模式和 PostgreSQL 可用性。
- `HEXCORE_MAX_ROOMS` 只限制 `active` 房间；归档房间不计入 active 上限。
- 归档房间拒绝新加入和所有 command，但允许裁判读取、导出和审计。
- 删除房间仅允许裁判或管理员执行，必须清理赛事状态、房间码摘要、session、事件、审计、检查点和 SSE 订阅。
- 房间码明文仍只在创建响应中显示一次；房间列表和管理接口不得返回房间码明文或摘要。
- 任意请求必须以 `tournamentId + sessionToken` 双重绑定校验，房间 A 的 session 不能读取或写入房间 B。
