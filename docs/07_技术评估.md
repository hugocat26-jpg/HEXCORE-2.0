# 技术评估

更新时间：2026-05-25

## 1. 当前技术选择

当前项目是静态前端 + 本地 Node 静态服务：

```text
原生 JavaScript
原生 CSS
本地 Node 静态服务
localStorage 状态持久化
自定义回归脚本 scripts/regression.js
内置 Browser / Word / Python 工具用于验收和文档 QA
```

当前阶段不引入后端、数据库或实时同步服务。系统以裁判端本地状态为权威，导入/导出 JSON 用于备份和迁移。

## 2. 是否需要 React 或构建框架

当前版本不需要立即引入 React、Vue、Vite 或 TypeScript 构建链。

原因：

- 已交付的裁判工作台主要复杂度在规则引擎、状态归一化和回归验证。
- 现有静态服务部署简单，适合线下裁判现场使用。
- 引入框架会改变构建、部署和回归路径，不应在当前稳定版本中作为默认改造。

未来若需要多人端、大屏端或复杂组件复用，可在保持引擎独立的前提下迁移前端框架。

## 3. 存储方案

当前阶段：

```text
localStorage + 状态导入/导出 JSON
```

当前要求：

- 状态快照、事件日志、撤销栈均可序列化。
- 导入状态必须经过 normalize。
- 未知字段不能破坏执行。
- 导出文件必须保留阵营、队长、金币、海克斯和事件日志。

未来多人阶段可迁移到：

```text
API + PostgreSQL + WebSocket
```

但现阶段不实现。

## 4. 主要技术风险

### 4.1 规则冲突风险

海克斯会同时影响顺位、商店、购买权、金币和入队结果。

当前控制手段：

- 顺位效果使用优先级。
- 入队统一经过 `assignment-engine` 同阵营、队长、容量校验。
- 复杂组合写入回归测试，例如海浪/大炮、海浪/背水、雪定/加急调货。

### 4.2 暗信息泄漏风险

`雪定饿的喵`、`骤雨 血雾 清风` 等效果涉及显示身份和真实身份分离。

当前裁判端可见真实状态，但仍必须在数据结构上保留 `displayPlayerId` 和真实 `playerId` 区分，为未来队长端隔离做准备。

### 4.3 状态恢复风险

本地刷新、导入恢复、撤销和异常修复可能导致队伍归属、购买权、海克斯状态或商店状态不一致。

当前控制手段：

- `Integrity Service` 检查跨阵营、重复归属、超员、禁用选手和卡池异常。
- 回归覆盖导入 XSS、路径穿越、状态结构异常和关键流程恢复。
- 所有高影响动作写事件日志。

### 4.4 文档漂移风险

项目经历过无金币轮抽、四轮金币抽卡、阵营锁定金币商店多个阶段。

当前控制手段：

- 现行开发手册以 `docs/06_开发计划.md` 为准。
- 现行规则规范以 `docs/14_阵营锁定10队金币商店模式_实施规范.md` 为准。
- 旧方案文档必须标注历史归档，不得作为当前规则直接执行。

## 5. 当前阶段结论

当前阶段定义为：

```text
裁判单端权威执行版本
```

技术目标不是重构框架，而是保持现场可运行、规则可解释、状态可恢复、回归可验证。未来多人端必须复用当前引擎边界，而不是绕过现有校验。
