# 技术评估

## 1. 当前阶段技术选择

建议当前阶段采用：

```text
Vite
TypeScript
原生 CSS
Vitest
Playwright
localStorage / IndexedDB
```

不建议第一阶段直接引入完整后端。当前更重要的是把规则引擎、事件模型和数据模型做对。

## 2. 是否需要 React

当前项目可以不立即引入 React。

原因：

- 核心复杂度在规则引擎，不在 UI。
- 裁判工作台页面数量有限。
- 过早引入框架会分散精力。

如果后续 UI 复杂度增加，可以再引入 React 或 Vue。架构上应保证引擎与 UI 解耦。

## 3. 存储方案

当前阶段：

```text
localStorage 或 IndexedDB
```

建议：

- 状态快照存在本地。
- 事件日志也存在本地。
- 导出 JSON 包含状态快照和事件日志。

未来多人阶段：

```text
PostgreSQL + API + WebSocket
```

当前应通过 `StorageService` 抽象存储，避免业务层直接调用 `localStorage`。

## 4. 主要技术风险

### 4.1 规则冲突风险

顺位类海克斯很多。如果没有统一优先级，会出现不可预测结果。

解决方案：

- 所有顺位效果使用 `priority`。
- 输出顺位解释。
- 对冲突规则写单元测试。

### 4.2 暗信息泄漏风险

致盲、雪定饿的喵、盲盒等效果涉及隐藏信息。

当前裁判代执行阶段风险较低，但未来队长端必须做到服务端级隔离。

解决方案：

- 从现在开始区分真实状态和角色视图。
- Visibility Engine 不能只做 CSS 模糊。

### 4.3 事件回放复杂度

盲盒转队、补偿回合、锁定契约会改变队伍和顺位。

解决方案：

- 所有动态变化都写事件。
- 不直接删除历史。
- 回退通过追加修正事件实现。

### 4.4 旧代码迁移风险

旧项目包含金币逻辑、5/7费口径和 1-4 档口径混用。

解决方案：

- 新代码不沿用 `cost` 命名。
- 旧数据导入时显式转换。
- 金币字段全部丢弃。

## 5. 实施建议

建议下一步不要直接迁移旧 `app.js`。

正确顺序：

1. 建立 TypeScript 类型。
2. 建立事件日志。
3. 建立引擎测试。
4. 实现最小轮抽闭环。
5. 再逐步迁移旧功能。

## 6. 当前阶段结论

当前阶段应定义为：

```text
裁判单端权威执行版本
```

不是多人实时版本。

但所有核心模块都应按未来多人模式设计边界：

- UI 不可信。
- 操作必须经过引擎校验。
- 状态变化必须事件化。
- 暗信息必须通过视图模型隔离。
- 存储必须可替换。

这样第一阶段能快速落地，未来队长自抽和裁判监督不会推倒重来。
