# 技术评估

更新时间：2026-05-30

## 1. 当前技术选择

当前项目同时维护裁判端本地控制台和多人端服务端权威副本：

```text
原生 JavaScript
原生 CSS
本地 Node 静态服务
localStorage 状态持久化
Node 多人端 API
SSE 实时同步
JSON / SQLite / PostgreSQL 持久化
自定义回归脚本 scripts/regression.js
内置 Browser / Word / Python 工具用于验收和文档 QA
```

裁判端维护版仍以本地状态为权威，导入/导出 JSON 用于备份和迁移。多人端以服务端权威状态为准，通过 command、event、projection 和 SSE 同步裁判端、队长端和观众端。

## 2. 是否需要 React 或构建框架

当前版本不需要立即引入 React、Vue、Vite 或 TypeScript 构建链。

原因：

- 已交付的裁判工作台主要复杂度在规则引擎、状态归一化和回归验证。
- 现有静态服务部署简单，适合线下裁判现场使用。
- 引入框架会改变构建、部署和回归路径，不应在当前稳定版本中作为默认改造。

未来若需要复杂组件复用，可在保持引擎独立的前提下评估 Vite + React 或 TypeScript。当前不建设大屏端。

## 3. 存储方案

裁判端维护版：

```text
localStorage + 状态导入/导出 JSON
```

当前要求：

- 状态快照、事件日志、撤销栈均可序列化。
- 导入状态必须经过 normalize。
- 未知字段不能破坏执行。
- 导出文件必须保留阵营、队长、金币、海克斯和事件日志。

多人端当前已支持：

```text
Node API + SSE + JSON 文件 / SQLite / PostgreSQL
```

PostgreSQL 是正式持久化方向；SQLite 和 JSON 文件适合本机或局域网试运行。

## 4. 主要技术风险

### 4.1 规则冲突风险

海克斯会同时影响顺位、商店、购买权、金币和入队结果。

当前控制手段：

- 顺位效果使用优先级。
- 入队统一经过 `assignment-engine` 同阵营、队长、容量校验。
- 复杂组合写入回归测试，例如海浪/大炮、海浪/背水、雪定/加急调货。

### 4.2 暗信息泄漏风险

`雪定饿的喵`、`骤雨 血雾 清风` 等效果涉及显示身份和真实身份分离。

当前裁判端可见真实状态，但仍必须在数据结构上保留 `displayPlayerId` 和真实 `playerId` 区分，为未来队长端隔离做准备。

### 4.3 状态恢复风险

本地刷新、导入恢复、撤销和异常修复可能导致队伍归属、购买权、海克斯状态或商店状态不一致。

当前控制手段：

- `Integrity Service` 检查跨阵营、重复归属、超员、禁用选手和卡池异常。
- 回归覆盖导入 XSS、路径穿越、状态结构异常和关键流程恢复。
- 所有高影响动作写事件日志。

### 4.4 文档漂移风险

项目经历过无金币轮抽、四轮金币抽卡、阵营锁定金币商店多个阶段。

当前控制手段：

- 当前待开发计划以 `docs/planning/当前待开发计划.md` 为准。
- 开发门禁以 `docs/planning/开发门禁.md` 为准。
- 现行规则规范以 `docs/product/阵营锁定10队金币商店模式实施规范.md` 为准。
- 旧方案资料应从源文档中移除，不得作为当前规则直接执行。

## 5. 当前阶段结论

当前阶段定义为：

```text
裁判端稳定维护 + 多人端服务端权威演进
```

技术目标不是一次性重构框架，而是保持现场可运行、规则可解释、状态可恢复、回归可验证。多人端必须复用当前引擎边界和服务端校验，而不是绕过现有规则。
