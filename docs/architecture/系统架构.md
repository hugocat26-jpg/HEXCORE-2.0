# 架构设计

更新时间：2026-05-30

当前状态：本文记录裁判端本地控制台和多人端服务端权威副本的架构边界。裁判端维护版保持稳定，多人端在 `codex/multiplayer-realtime` 分支持续演进。

## 1. 当前阶段架构

裁判端维护版是本地单端应用：

```text
裁判浏览器
  ├─ UI 页面
  ├─ 应用状态 AppState
  ├─ 规则引擎
  ├─ 事件日志
  └─ 本地存储
```

裁判端执行所有本地操作。多人端不直接复用本地状态作为权威，而是通过服务端 command 和投影同步。

## 2. 多人端架构边界

多人端当前只包含裁判端、队长端和观众端，不建设大屏端：

```text
客户端
  ├─ 裁判端
  ├─ 队长端
  └─ 观众端

服务端
  ├─ 鉴权与 session
  ├─ 权限校验
  ├─ 权威规则执行
  ├─ 公开投影
  ├─ 事件流 SSE
  ├─ 房间生命周期管理
  └─ JSON / SQLite / PostgreSQL 存储
```

队长端和观众端只能消费按角色裁剪后的投影；任何写入都必须由服务端校验角色、队伍、回合、版本和规则窗口。

## 4. 单向数据流

```text
裁判操作
  ↓
Action
  ↓
Engine 校验并计算结果
  ↓
生成 Domain Event
  ↓
更新 AppState
  ↓
保存 Event Log / Snapshot
  ↓
UI 重新渲染
```

禁止 UI 直接改 `players`、`captains`、`draft` 等核心状态。

## 5. 引擎职责

### Draft Engine

- 控制流程状态机。
- 判断当前阶段允许哪些操作。
- 处理开始金币商店抽选、进入下一位、进入下一轮、完成四轮抽选。

### Turn Order Engine

- 生成基础蛇形顺位。
- 应用顺位类海克斯。
- 处理顺位冲突和优先级。

### Pool Engine

- 按规则模式、队长锁定、禁用状态、出勤状态和归属状态计算有效候选池。
- 双阵营模式按本地/外地阵营隔离；无阵营模式使用公共候选池。
- 按费用分档和官方成绩来源输出 1-5 费卡池。
- 保证队长不进入购买、补位或海克斯入队候选。

### Probability Engine

- 从有效卡池中按当前轮金币商店概率计算候选。
- 应用空费用等比放大、保底、排除和特殊概率。
- 生成可记录的随机结果。

### Hexcore Engine

- 读取海克斯定义。
- 识别触发时机。
- 调用对应 effect handler。
- 生成运行时效果。

### Visibility Engine

- 控制卡片身份是否可见。
- 支持致盲、暗牌、身份互换、延迟揭示。
- 裁判端和多人端都需要该模块，用于隔离暗牌、身份互换和角色投影。

### Assignment Engine

- 处理购买入队、海克斯入队、最终随机补位和替补替换。
- 双阵营模式执行同阵营、非队长、非禁用、容量和重复归属硬校验；无阵营模式跳过阵营校验。
- 处理同阵营置换、公共池置换与海克斯补偿。
- 队伍满员后跳过后续选人。

## 6. 当前阶段权限边界

多人端当前角色：

```text
referee
captain
viewer
```

权限边界：

- `referee`：最高权限，可创建、归档、删除房间，执行抽卡、赛程、替补和导出。
- `captain`：只在本人回合或本人海克斯窗口内写入；其他队长回合跟随当前队长视角只读。
- `viewer`：始终只读，只能查看当前回合队长视角和公开信息。

未来角色预留：

```text
super_admin
tournament_admin
referee
supervisor
captain
viewer
```

所有写操作仍必须经过服务端 command 权限校验。前端隐藏按钮只用于改善体验，不能作为安全边界。

## 7. 多房间与 PostgreSQL 边界

- 每个房间以 `tournamentId` 隔离权威状态、房间访问摘要、session、事件、审计、检查点和 SSE 订阅。
- 任意读写请求都必须绑定 `tournamentId + sessionToken`；房间 A 的 session 不能读取或写入房间 B。
- `HEXCORE_MAX_ROOMS` 只限制 `active` 房间数；归档房间不计入上限。
- `active` 房间可加入和写入；`archived` 房间只读；`deleted` 房间从服务端索引中清除。
- PostgreSQL store 持久化 `room_status`、`archived_at`、`deleted_at`、`deleted_by` 和 `last_room_updated_at`，用于重启恢复房间列表和生命周期状态。
- PostgreSQL 模式下，跨实例事件通过轮询 `hexcore_events` 补偿给本实例 SSE 订阅者，仍走公开投影，不直接暴露私有事件。
