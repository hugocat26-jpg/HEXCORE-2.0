# HEXCORE2.0 协作钩子

本项目所有回复、解释、代码注释均使用简体中文。

## 严格任务执行循环钩子

`post-task` 是本项目唯一任务执行循环钩子。它的目的不是简单汇报，而是强制 Codex 按任务文档严格推进、持续验证、持续修复，直到所有计划和验收标准全部完成。

循环必须有上限，禁止无限重试消耗上下文和 token。同一任务文档连续 `incomplete` 默认最多 5 次；达到上限后必须停止自动执行，汇总剩余问题、阻断原因和建议下一步，等待用户确认。确需调整时使用 `--max-attempts=<次数>`，但不得为了绕过复核而随意调高。

每次完成一个阶段后，必须执行以下后置流程：

1. 运行本地门禁：
   - 普通任务运行 `npm run post-task -- --status=<complete|incomplete>`
   - 若任务有专用文档，运行 `npm run post-task -- --doc=<任务文档路径> --status=<complete|incomplete>`
   - 若需要临时调整循环上限，追加 `--max-attempts=<次数>`；默认上限为 5 次连续 `incomplete`。
   - 钩子会自动读取任务文档、运行 `npm test`、运行 `git diff --check`，并提取验收项、推荐顺序和阻断线索。
2. 检查当前计划是否完成：
   - 若任务文档仍存在未勾选清单、阻断线索、待处理、尚未完成、未完成、失败、TODO、FIXME、等待用户、待验证、待修复等线索，视为 `incomplete`，必须继续执行下一项，不允许只汇报状态后停止。
   - 若本轮计划已完成，先按改动风险判断是否需要追加审查；低风险文档、样式或钩子整理不默认做冗余全量安全扫描。
   - 只有涉及导入/导出、本地存储、静态服务、状态归一化、删除重置、依赖、命令执行、鉴权或其他信任边界时，才使用 Codex Security 的 `security-scan` 或差异范围审查。
   - 对必要安全审查发现的问题，使用 Codex Security 的 `fix-finding` 技能修复并重新验证。
   - 修复和验证通过后，如本轮属于较大改动，提交并推送到 Gitee。
3. 若本轮计划未完成：
   - 继续执行开发计划中的下一步。
   - UI、交互、浏览器验证相关任务在实际改动渲染、布局或关键交互时，使用 Build Web Apps 的前端调试/测试流程。
   - 较大功能点完成后，只有涉及安全敏感代码或用户可控输入边界时，再使用 Codex Security 技能做代码审查和必要修复。
   - 执行完成后再次运行 `post-task`，直到 `--status=complete` 通过或连续 `incomplete` 达到上限。
   - 达到上限时不得继续自动重试，必须先向用户汇报剩余计划项、失败原因和下一步建议。
4. 推送约束：
   - 仅在改动较大、阶段性闭环完成、或用户明确要求时推送。
   - 推送前必须再次通过 `npm test` 和 `git diff --check`。
   - 不提交无关未跟踪文件，尤其是临时设计文档或本地实验产物。

## 完成判定

`complete` 表示当前开发计划的验收项已全部满足，并且本地门禁通过。钩子会把它视为“申请完成”；若任务文档中仍存在未勾选清单、环境阻断、待处理、尚未完成、未完成、等待用户、失败、TODO、FIXME、待验证、待修复等线索，钩子必须失败退出。
`incomplete` 表示还有计划项未完成，Codex 必须继续按开发计划推进下一项，并在下一阶段完成后再次运行 `post-task`，但连续 `incomplete` 不能超过钩子上限。

如果完成状态无法从代码和文档中可靠判断，默认按 `incomplete` 处理，继续执行下一项，而不是提前推送。
